    1. Create `packages/api/src/services/queue.service.ts` following the post.service.ts pattern:
       - `createQueue(db: Db, userId: string, input: CreateQueueInput): Promise<Queue>` -- insert into queues table, calculate initial nextRunAt using `calculateNextRunAt` from schedule-evaluation, return the created queue row
       - `updateQueue(db: Db, userId: string, queueId: string, input: UpdateQueueInput): Promise<Queue>` -- update queue row, recalculate nextRunAt, return updated row
       - `deleteQueue(db: Db, userId: string, queueId: string): Promise<void>` -- delete queue row (posts get queue_id set null via ON DELETE SET NULL)
       - `getQueues(db: Db, userId: string, filters: QueueQueryInput): Promise<QueueListItem[]>` -- query queues with optional network/status filters. Join with socialProfiles for network type. Include aggregate: total post count (COUNT of posts with matching queue_id), last published (MAX published_at of queue posts). Return: id, name, profileId, profileName, network, postCount, isPaused, isRecycling, lastPublishedAt, nextRunAt, cursorPosition, seasonalStart, seasonalEnd, hourSlots, daysOfWeek, notes
       - `getQueueById(db: Db, userId: string, queueId: string): Promise<Queue | null>` -- single queue with all fields, filtered by userId
       - `copyQueueConfig(db: Db, userId: string, queueId: string): Promise<QueueConfig>` -- return the schedule configuration fields only (no id, no state fields) for pre-populating a new queue form
       - `addPostToQueue(db: Db, userId: string, queueId: string, postId: string): Promise<void>` -- set post's queue_id and assign next available position: `SELECT COALESCE(MAX(queue_position), 0) + 1 FROM posts WHERE queue_id = ?`; set post status to 'queued' using `transitionPost()` if currently in draft state
       - `removePostFromQueue(db: Db, userId: string, queueId: string, postId: string): Promise<void>` -- set queue_id and queue_position to null
       - `getQueuePosts(db: Db, userId: string, queueId: string): Promise<QueuePost[]>` -- list posts in queue ordered by queue_position ASC, include: id, text (first 80 chars truncated), status, hasSpinnableText, autoDestructAfter, queuePosition, platformPostId, publishedAt
       - `movePostUp(db: Db, userId: string, queueId: string, postId: string): Promise<void>` -- per RESEARCH.md Pattern 4: transaction-based position swap with the previous post. Verify post belongs to this queue and userId owns the queue.
       - `movePostDown(db: Db, userId: string, queueId: string, postId: string): Promise<void>` -- same as moveUp but swaps with the next post
       All functions MUST filter by `userId` to enforce queue ownership (T-05-01-02).

    2. Create `packages/api/src/services/auto-destruct-queue.service.ts`:
       - `createAutoDestructQueueService(redis: Redis): AutoDestructQueueService`
       - Interface: `{ enqueueAutoDestruct(postId: string, platformPostId: string, publishedAt: Date, autoDestructAfter: string): Promise<void> }`
       - Calculates delay: `Math.max(0, targetTime - Date.now())` where targetTime = publishedAt + parsed duration (D-14)
       - Uses `buildAutoDestructJobId(postId, platformPostId)` for jobId deduplication
       - Job payload: `{ postId, platformPostId, correlationId }` -- platformPostId in payload, NOT re-read from DB at execution time (RESEARCH.md Pitfall 1)

    3. Create `packages/api/src/routes/queues.ts` following the posts router factory pattern:
       ```typescript
       interface QueuesDependencies { db: Db; autoDestructQueueService?: AutoDestructQueueService; }
       export function createQueuesRouter(deps: QueuesDependencies): Router;
       ```
       Routes (all behind `requireAuth`):
       - `GET /api/queues` -- list queues with queueQuerySchema validation on query params
       - `POST /api/queues` -- create queue with createQueueSchema body validation
       - `GET /api/queues/:id` -- get single queue (validateUuidParam('id'))
       - `PUT /api/queues/:id` -- update queue with updateQueueSchema body validation
       - `DELETE /api/queues/:id` -- delete queue (returns 204)
       - `GET /api/queues/:id/config` -- get queue config for copy (returns schedule fields only)
       - `GET /api/queues/:id/posts` -- list queue posts
       - `POST /api/queues/:id/posts` -- add existing post to queue (body: { postId })
       - `POST /api/queues/:id/posts/:postId/move-up` -- reorder up
       - `POST /api/queues/:id/posts/:postId/move-down` -- reorder down
       - `DELETE /api/queues/:id/posts/:postId` -- remove post from queue

       Error handling:
       - Queue not found or ownership mismatch: 404 (timing-safe, no info leak per T-05-01-02)
       - Invalid input: 400 with Zod error
       - Post already in another queue (D-08): 409 Conflict with message "Post is already assigned to another queue"

    4. Mount the router in `packages/api/src/app.ts`:
       - Add `import { createQueuesRouter } from './routes/queues.js';`
       - Mount: `app.use('/api/queues', createQueuesRouter({ db, autoDestructQueueService }))`
       - The autoDestructQueueService is created from Redis connection similar to how publishQueueService is created

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> API | All user input crosses here; validated via Zod |
| API -> DB | Ownership enforced via userId WHERE clause on every query |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Tampering | Queue CRUD inputs | mitigate | Zod schema validates all inputs; hourSlots range 6-23, daysOfWeek 0-6, intervalValue 1-999 |
| T-05-02-02 | Elevation of Privilege | Queue ownership bypass | mitigate | All service functions filter by `eq(queues.userId, userId)` from session; 404 on mismatch (no info leak) |
| T-05-02-03 | Tampering | Post reorder across queues | mitigate | moveUp/moveDown verify the post's queueId matches the route param queueId within the transaction |
| T-05-02-04 | Information Disclosure | Queue enumeration via ID | mitigate | UUID primary keys + userId ownership filter; 404 for all non-owned resources regardless of existence |
| T-05-02-05 | Tampering | Auto-destruct job injection | mitigate | Auto-destruct jobs only enqueued by worker/scanner, never directly from user API input. The API service is used internally by the publish worker. |
</threat_model>
