## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker -> Twitter API | Decrypted credentials cross here; never logged, never cached |
| worker -> DB | Post state transitions enforced via transitionPost() |
| BullMQ job payload | Contains platformPostId for deletion -- must match what was published |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Information Disclosure | Credential exposure in auto-destruct | mitigate | Same credential discipline as publish: decrypt in function scope, const binding, never logged. twitter-delete.service.ts mirrors twitter-publish.service.ts pattern exactly |
| T-05-03-02 | Tampering | Auto-destruct job payload manipulation | mitigate | Auto-destruct jobs enqueued only by publish worker or scanner, never from user API. Job payload contains platformPostId captured at publish time, not user-supplied |
| T-05-03-03 | Denial of Service | Queue scanner overwhelming publish queue | mitigate | nextRunAt update within scanner transaction prevents re-evaluation on next tick (Pitfall 2); BullMQ jobId dedup prevents duplicate jobs |
| T-05-03-04 | Tampering | Wrong tweet deleted via stale platformPostId | mitigate | platformPostId comes from job payload captured at publish time, NOT re-read from DB (RESEARCH.md Pitfall 1). Each publish cycle creates a new auto-destruct job with its own platformPostId |
</threat_model>
