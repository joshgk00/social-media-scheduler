## Warnings

### WR-01: Queue-empty notifications fire every 60s for persistently empty queues

**File:** `packages/worker/src/queue-scanner.ts:268-282`
**Issue:** When a queue has no posts, the transaction returns null and the code sends a `queue_empty` notification. However, `nextRunAt` is only updated inside the transaction when a post IS found (line 240-262). For an empty queue, `nextRunAt` stays in the past, so the queue passes the `lte(queues.nextRunAt, sql'NOW()')` filter on every 60s scan. The `notificationThrottle` map only records a timestamp on notification enqueue *failure* (inside `.catch()`), not on success. This means a successful queue-empty notification does not set the throttle, and another notification fires 60 seconds later. A persistently empty queue will generate a queue-empty notification every single scan cycle indefinitely.

**Fix:** Either throttle on success too, or update `nextRunAt` when the queue is empty to back off. The simplest fix is to set the throttle timestamp after a successful notification:

```typescript
await notificationQueue.add(JOB_NAMES.queueEmptyNotification, {
  kind: 'queue_empty',
  queueId: queue.id,
  queueName: queue.name,
  at: new Date().toISOString(),
}).then(() => {
  notificationThrottle.set(queue.id, Date.now());
}).catch((err: unknown) => {
  notificationThrottle.set(queue.id, Date.now());
  logger.error({ err, queueId: queue.id }, 'Failed to enqueue queue-empty notification');
});
```

Also consider a longer throttle window (e.g., 30 minutes) for successful empty-queue notifications since the condition is unlikely to self-resolve in 60 seconds.

## Info

### IN-01: Decorative badge dots missing aria-hidden on Paused and Active states

**File:** `packages/web/src/components/queues/QueueStatusBadge.tsx:45,60`
**Issue:** The "Seasonal pause" badge correctly marks its decorative dot with `aria-hidden="true"` (line 35), but the "Paused" badge (line 45) and "Active" badge (line 60) omit `aria-hidden` on their dots. Screen readers may announce these as empty elements.
**Fix:** Add `aria-hidden="true"` to the dot spans on lines 45 and 60 for consistency with line 35.

### IN-02: Notification throttle window is only 60 seconds

**File:** `packages/worker/src/queue-scanner.ts:270`
**Issue:** The throttle check uses `60_000` ms (60 seconds), which matches the scan interval. Even after fixing WR-01 to throttle on success, a 60-second throttle means notifications would fire every other scan cycle for an empty queue. Consider whether a longer window (e.g., 15 or 30 minutes) better matches user expectations.
**Fix:** Extract the throttle duration to a named constant and consider a longer interval:

```typescript
const QUEUE_EMPTY_NOTIFICATION_THROTTLE_MS = 30 * 60_000; // 30 minutes
```
