## Architecture Patterns

### WR-01: Transaction Boundary Fix

**What:** The recycling bulk update + MIN(queue_position) select already run inside `db.transaction()`. But after the transaction returns, the cursor advance (`db.update(queues).set({ cursorPosition, nextRunAt })`) runs as a separate query outside the transaction. If the process crashes between BullMQ enqueue and cursor advance, the cursor is stuck at the old position. BullMQ jobId dedup prevents double-publish, but the queue stalls until manual intervention.

**Current code flow (queue-scanner.ts lines 166-299):**
```
1. db.transaction: find next post (or recycle + find)  <-- inside tx
2. publishQueue.add(...)                                <-- outside tx (correct -- BullMQ is Redis, not Postgres)
3. db.update(queues).set({ cursorPosition, nextRunAt }) <-- outside tx (BUG)
```

**Fix:** Move the cursor advance INTO the transaction from step 1. The transaction should return the candidate post AND advance the cursor atomically. BullMQ enqueue stays outside the transaction (Redis operations must not be inside a Postgres transaction).

**Corrected flow:**
```
1. db.transaction:
   a. find next post (or recycle + find)
   b. advance cursor + update nextRunAt         <-- moved into tx
2. publishQueue.add(...)                         <-- outside tx (unchanged)
```

**Edge case:** If BullMQ enqueue fails after the cursor was advanced in the transaction, the cursor has moved past a post that was never enqueued. This is acceptable because: (a) the post remains in `queued` status so it can be picked up on the next cycle if the cursor wraps, and (b) this failure mode is less harmful than the current bug where cursor stalls indefinitely. [VERIFIED: codebase analysis]

**Alternative considered:** Keep cursor advance outside tx but add a recovery mechanism. Rejected -- transaction is simpler and the standard pattern for multi-table DB consistency.

### WR-04: QueueStatusBadge Seasonal Logic Fix

**What:** The frontend `isInSeasonalPause()` function in `QueueStatusBadge.tsx` has inverted logic for cross-year windows (e.g., Nov 1 - Jan 31). The function returns "paused" when `today > seasonalEnd && today < seasonalStart`, which for Nov-Jan means Feb-Oct is flagged as paused (wrong) and Nov-Jan is flagged as active (wrong).

**Backend is correct:** The shared `isWithinSeasonalWindow()` in `packages/shared/src/lib/schedule-evaluation.ts` handles cross-year windows correctly with `currentMonthDay >= startMonthDay || currentMonthDay <= endMonthDay`. The bug is ONLY in the frontend component's duplicate logic. [VERIFIED: codebase reading]

**Current buggy code (QueueStatusBadge.tsx lines 19-24):**
```typescript
if (seasonalStart <= seasonalEnd) {
  // Same-year window: paused when outside range
  return today < seasonalStart || today > seasonalEnd;  // CORRECT
}
// Cross-year: paused when today is in the gap (after end AND before start)
return today > seasonalEnd && today < seasonalStart;    // INVERTED
```

For Nov-Jan window: `seasonalStart='11-01', seasonalEnd='01-31'`. Since `'11-01' > '01-31'`, the cross-year branch executes. On July 15 (`today='07-15'`): `'07-15' > '01-31' && '07-15' < '11-01'` = true, so it shows "Seasonal pause". But July IS the active gap between seasonal windows, and the queue IS in seasonal pause during Nov-Jan.

**Wait -- re-analyzing the semantics:** The seasonal window defines when the queue RUNS (is active). Outside the window, the queue is paused. For a Nov-Jan seasonal window: Nov-Jan = active, Feb-Oct = paused. The function `isInSeasonalPause` should return true when we're OUTSIDE the seasonal window.

For cross-year window `seasonalStart='11-01', seasonalEnd='01-31'`:
- Active months: Nov, Dec, Jan (within window)
- Paused months: Feb-Oct (outside window)

Current code returns `today > seasonalEnd && today < seasonalStart` for cross-year. On July 15: `'07-15' > '01-31' && '07-15' < '11-01'` = true = paused. That IS correct for "is paused".

Let me re-read the audit finding more carefully.

**Audit says:** "QueueStatusBadge cross-year seasonal window logic inverted -- shows 'Seasonal pause' during active months for Nov-Jan windows."

Let me trace through a November date to verify. On Nov 15 (`today='11-15'`): `'11-15' > '01-31' && '11-15' < '11-01'` = `true && false` = false. Not paused. That's correct -- Nov is within the active window.

On Feb 15 (`today='02-15'`): `'02-15' > '01-31' && '02-15' < '11-01'` = `true && true` = true. Paused. Correct -- Feb is outside the window.

On Jan 15 (`today='01-15'`): `'01-15' > '01-31'` = false. Not paused. Correct -- Jan is within window.

On Dec 15 (`today='12-15'`): `'12-15' > '01-31' && '12-15' < '11-01'` = `true && false` = false. Not paused. Correct -- Dec is within window.

**The current cross-year logic appears correct.** The audit report may have the bug description inverted, OR the audit is referring to a different interpretation of "seasonal window."

Let me check whether the seasonal window means "queue runs during this period" or "queue pauses during this period." The backend `isWithinSeasonalWindow` returns true = "queue should run." The scanner CONTINUES (skips the queue) when `!isWithinSeasonalWindow()`. So the seasonal window defines the ACTIVE period.

For the badge: `isInSeasonalPause` should return true when we're outside the active period. The same-year branch: `today < seasonalStart || today > seasonalEnd` -- outside the [start, end] range. Correct. The cross-year branch for the gap (Feb-Oct when window is Nov-Jan): `today > seasonalEnd && today < seasonalStart` -- in the gap between end and start. This IS the outside-of-window period. **This appears correct.**

However, the audit was performed by automated code analysis. Let me check if there's a mismatch between the two implementations' definition of "within" vs "paused." The backend uses `>=` and `<=` (inclusive boundaries) while the frontend uses `<` and `>` (exclusive). On exact boundary dates, there could be an off-by-one, but that's not the "inverted" bug described.

Let me re-examine: the audit says "shows 'Seasonal pause' during Feb-Oct (the active gap)". Wait -- for a Nov-Jan window, Feb-Oct IS the gap (paused period), not the active period. The audit description itself says "active gap" which is contradictory. I think the audit report has a confusing description, and the actual complaint may be about the INTERPRETATION of seasonal windows being backwards.

**Actually, upon even closer re-reading of the audit:** "For Nov-Jan seasonal windows, the badge shows 'Seasonal pause' during Feb-Oct (the active gap) and 'Active' during Nov-Jan (the actual pause)." This implies that Nov-Jan IS the pause period, not the active period. If that's the semantic intent -- seasonal window = pause window, not active window -- then the frontend is wrong because it treats the window as an active window.

But the BACKEND treats the seasonal window as the ACTIVE window (scanner runs during the window). So there's a semantic conflict. The backend and frontend disagree on what "seasonal window" means.

**Resolution:** The backend's `isWithinSeasonalWindow` defines the window as the ACTIVE period (queue runs during this window). The frontend badge should align. If the badge shows "Seasonal pause" when outside this window, it's correct. The audit report appears to have an incorrect description of the bug.

However, since the audit explicitly lists WR-04 as a gap-closure item and this phase's success criteria say "QueueStatusBadge seasonal pause shows correctly for Nov-Jan windows year-round," we should add comprehensive tests to prove correctness (or find the actual bug through testing). If tests pass with current logic, WR-04 may already be correct and just needs test coverage.

**Fix approach:** Write comprehensive component tests for QueueStatusBadge covering all seasonal window scenarios (same-year, cross-year, boundary dates, Feb date during Nov-Jan window, July date during Nov-Jan window, etc.). If any test fails, fix the logic. If all pass, document that WR-04 was a false positive in the audit.

### IN-03: Module-Scope Logger Fix

**What:** In `queue-scanner.ts`, line 87 calls `createLogger('queue-scanner')` inside the `evaluateQueues()` function body. This function runs every 60 seconds. While `createLogger` is lightweight (just creates a pino child logger), allocating a new logger object 1,440 times/day is wasteful and non-idiomatic.

**Fix:** Move `const logger = createLogger('queue-scanner')` to module scope (top of file, after imports). The `startQueueScanner` function already has its own module-scope-like logger on line 321; that one is fine since `startQueueScanner` runs once.

**Note:** The function signature `evaluateQueues(db, publishQueue, ...)` is exported and tested. Moving logger to module scope doesn't change the function signature or behavior, only the allocation pattern.

### Anti-Patterns to Avoid

- **BullMQ inside Postgres transaction:** Never put Redis/BullMQ calls inside a `db.transaction()` callback. If the transaction rolls back, the BullMQ job persists in Redis, causing inconsistency. Keep BullMQ operations outside the DB transaction boundary.
- **Duplicating shared logic in frontend:** The `isInSeasonalPause` in `QueueStatusBadge.tsx` duplicates `isWithinSeasonalWindow` from `@sms/shared`. Consider importing the shared function instead. However, `@sms/shared` uses Luxon's `DateTime` which adds weight to the frontend bundle -- so the lighter string-based approach in the component is acceptable if kept correct.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Transaction management | Manual two-step DB updates | `db.transaction()` from drizzle-orm | Guarantees atomicity, automatic rollback on error |
| Date/seasonal logic | Separate frontend/backend implementations | Import from `@sms/shared` or keep both tested | Divergent implementations will diverge in bugs |

## Common Pitfalls

### Pitfall 1: BullMQ Enqueue in DB Transaction
**What goes wrong:** Putting `publishQueue.add()` inside `db.transaction()` means if the transaction rolls back, the BullMQ job still exists in Redis.
**Why it happens:** Natural instinct to wrap everything in one atomic operation.
**How to avoid:** Keep BullMQ calls AFTER the transaction commits. Use jobId dedup as a safety net.
**Warning signs:** Redis jobs processing for posts that don't exist or have wrong status.

### Pitfall 2: String-Based Date Comparison for Seasonal Windows
**What goes wrong:** Lexicographic string comparison works for MM-DD format but fails if format changes.
**Why it happens:** `'02-15' < '11-01'` works because month comes first.
**How to avoid:** Use numeric comparison (month * 100 + day) like the backend does, or import the shared utility.
**Warning signs:** Seasonal badges showing wrong state around month boundaries.

### Pitfall 3: Mock Drift in Queue Scanner Tests
**What goes wrong:** The queue scanner tests mock all schedule evaluation functions. If the real implementations change behavior, tests still pass with stale mocks.
**Why it happens:** Necessary isolation for unit tests, but creates a maintenance burden.
**How to avoid:** Add integration-level tests that use real schedule evaluation functions alongside the mocked unit tests.
**Warning signs:** Tests pass but real behavior differs.
