# Phase 06.3 Plan 01: Queue Scanner WR-01 and IN-03 Fixes Summary

Atomic cursor advance inside db.transaction and single module-scope logger allocation for queue-scanner.

## What Changed

### WR-01: Cursor advance moved into transaction
- Post selection, recycling, and cursor advance now execute atomically inside a single `db.transaction()` call
- If the process crashes mid-transaction, both post selection and cursor advance roll back together -- no stuck-queue state
- BullMQ enqueue stays outside the transaction (Redis operations must not be inside a Postgres transaction)
- If BullMQ enqueue fails after the transaction commits, the post remains `queued` and will be picked up on the next cycle when the cursor wraps; BullMQ jobId dedup prevents double-publish

### IN-03: Logger moved to module scope
- `createLogger('queue-scanner')` called once at module scope instead of inside `evaluateQueues` (line 87) and `startQueueScanner` (line 321)
- Eliminates 1,440 unnecessary logger object allocations per day (one per 60-second tick)
