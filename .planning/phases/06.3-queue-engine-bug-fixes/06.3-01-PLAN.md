**WR-01 -- Cursor advance into transaction:**
1. Change the transaction at line 166 to return `{ post, cursorData }` instead of just the post. The transaction return type becomes `Promise<{ post: QueuedPostRow; newCursorPosition: number } | null>`.
2. Move the `db.update(queues).set({ cursorPosition, nextRunAt, updatedAt })` block (lines 277-299) INTO the transaction callback, using `tx.update()` instead of `db.update()`. The `calculateNextRunAt()` call and its arguments stay the same.
3. After the transaction, the code that calls `publishQueue.add()` (lines 262-274) stays OUTSIDE the transaction -- BullMQ/Redis must never be inside a Postgres transaction.
4. Remove the post-enqueue `db.update(queues)` call (lines 277-299) since cursor advance now happens inside the transaction.
5. Update the post-transaction code to use `result.post` where it previously used `nextQueuedPost`.

The corrected flow:
```
1. db.transaction:
   a. find next post (or recycle + find)
   b. tx.update(queues).set({ cursorPosition: candidate.queuePosition, nextRunAt, updatedAt })
   c. return { post: candidate, newCursorPosition: candidate.queuePosition }
2. publishQueue.add(...)  // outside tx, unchanged
```

Edge case note: If BullMQ enqueue fails after cursor advance committed in the transaction, the post stays `queued` and will be picked up on a future cycle when cursor wraps. This is safer than the current bug where cursor stalls indefinitely.
