**Analog:** Same file, existing transaction block

**Code to move INTO the transaction** (lines 277-299 -- cursor advance currently outside tx):
```typescript

    cursorPosition: nextQueuedPost.queuePosition,
```

**Frontend implementation under test** (QueueStatusBadge.tsx lines 12-24):
```typescript
function isInSeasonalPause(seasonalStart?: string | null, seasonalEnd?: string | null): boolean {
  if (!seasonalStart || !seasonalEnd) return false;
  const now = new Date();
  const month = String(now.getMonth() + 1).padStart(2, '0');
  const day = String(now.getDate()).padStart(2, '0');
  const today = `${month}-${day}`;

  if (seasonalStart <= seasonalEnd) {
    return today < seasonalStart || today > seasonalEnd;
  }
  // Cross-year: paused when today is in the gap (after end AND before start)
  return today > seasonalEnd && today < seasonalStart;
}
```

**Fix approach:** Research analysis suggests the logic may be correct, but boundary comparisons use `<`/`>` (exclusive) where the backend uses `>=`/`<=` (inclusive). Write comprehensive tests first. If tests reveal a bug, fix it. If all pass, document WR-04 as a false positive. The `isInSeasonalPause` function needs to be testable -- it currently reads `new Date()` internally. To test deterministically, either:
1. Extract the date as an optional parameter (preferred, matches backend pattern)
2. Use `vi.useFakeTimers()` to control `new Date()`
