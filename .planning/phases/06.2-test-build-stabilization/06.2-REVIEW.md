## Summary

The migration runner (`migrate.ts`) is well-structured and clearly thought through — advisory lock, per-migration transactions, SAVEPOINT per statement, hash-based idempotency. The test suite mirrors that design rigour: snapshot drift detection, staged-migration file swap, crash-safe SIGINT handler. No fundamental design problems.

One critical SQL injection issue exists in the `INSERT INTO __drizzle_migrations` statement. Three warnings cover a shared module-level client in the test file (inhibits parallel test runs), a missing `error` field on the `MigrationLogger` interface, and a `require()` call inside an ESM module. Two info items cover coverage threshold comments and a type annotation.

## Critical Issues

### CR-01: SQL injection via unparameterized hash and `entry.when` in INSERT

**File:** `packages/db/src/migrate.ts:202`
**Issue:** The journal row insert is built by string interpolation:
```ts
`INSERT INTO "drizzle"."__drizzle_migrations" (hash, created_at) VALUES ('${hash}', ${entry.when})`
```
`hash` is a hex string derived from `createHash`, so it is safe in practice. However `entry.when` is read directly from the JSON journal file — if that file is tampered with or malformed (e.g., a non-numeric string), the interpolated value is injected verbatim into the SQL. Additionally, using unparameterized literals is a structural violation of the project's convention and sets a dangerous precedent in migration code. Any future edit that touches this line may introduce an actual injection vector.

**Fix:**
```ts
await client.unsafe(
  `INSERT INTO "drizzle"."__drizzle_migrations" (hash, created_at) VALUES ($1, $2)`,
  [hash, entry.when],
);
```
The `postgres` driver's tagged-template syntax is unavailable on `ReservedSql.unsafe()`, but `unsafe(sql, params)` accepts a second positional-params array and emits a parameterized query.

## Warnings

### WR-01: Module-level shared client in `migrate.test.ts` inhibits parallel test isolation

**File:** `packages/db/src/__tests__/migrate.test.ts:19`
**Issue:** `testClient` is created at module scope and reused across all six `describe` blocks:
```ts
const testClient = makeTestClient();
```
Vitest can shard or parallelize at the file level, but more importantly, any test that crashes without reaching `afterAll` will leave the connection open and leak to the next run. The `closeTestClient` call in `afterAll` is the only cleanup path — there is no SIGINT or exit handler for it, unlike the file-swap restoration logic. If a test runner hard-kills the worker, the connection leaks.

More practically, concurrent-caller test (Scenario 6) races two `runMigrations` calls that each create their own internal clients. The `testClient` adds a third open connection from the same test file, pushing the `max: 1` constraint comment in `test-db.ts` into misleading territory: `testClient` itself also uses `max: 1`, but there are now two simultaneous clients (testClient + runMigrations internal). Under pool exhaustion these contend.

**Fix:** Create and close the client per `describe` block using `beforeAll`/`afterAll`, or at minimum register a `process.once('exit')` handler in `test-db.ts` that calls `client.end()` synchronously (parallel to the `restoreAllBackups` pattern already in the test).

### WR-02: `require('node:fs')` inside an ESM module

**File:** `packages/db/src/__tests__/migrate.test.ts:103`
**Issue:**
```ts
// eslint-disable-next-line @typescript-eslint/no-require-imports
const fs = require('node:fs');
```
The package is declared `"type": "module"` in `packages/db/package.json`, which means all `.ts` files compile to ESM. `require()` is not available in native ESM at runtime. This currently works because `tsx` (or Vitest's transform) handles the interop, but it is fragile: if the test runner config or the TypeScript target changes, this will break with `ReferenceError: require is not defined`.

The comment admits the violation with an eslint-disable; the underlying need (synchronous file I/O in a process exit handler) is real, but the solution should use the Node.js built-in `fs.writeFileSync` imported as an ESM named import rather than a runtime `require`.

**Fix:**
```ts
import { writeFileSync } from 'node:fs';

// Inside restoreAllBackups():
writeFileSync(backup.path, backup.originalContent);
```
`writeFileSync` is synchronous and available as a named ESM import from `node:fs`. Remove the `require()` call entirely.

### WR-03: `MigrationLogger` interface has no `error` level — unlock failures log at `warn`

**File:** `packages/db/src/migrate.ts:111-114`
**Issue:** When `pg_advisory_unlock` throws, the failure is logged at `warn` level:
```ts
logger.warn('Failed to release migration advisory lock; session close will release it', {
  error: (unlockErr as Error).message,
});
```
A failure to release an advisory lock is operationally significant — it means every subsequent migration attempt on any other caller will block until the session is reaped by Postgres. This is not a warning; it is an error that operators need to be paged on. The `MigrationLogger` interface (line 55-58) only exposes `info` and `warn`, so there is no way to emit at `error` without changing the interface. In the interim, callers using structured logging pipelines (pino, Datadog) will miss this as an alert trigger.

**Fix:** Add `error` to the interface and use it:
```ts
interface MigrationLogger {
  info(msg: string, ctx?: Record<string, unknown>): void;
  warn(msg: string, ctx?: Record<string, unknown>): void;
  error(msg: string, ctx?: Record<string, unknown>): void;
}

const defaultLogger: MigrationLogger = {
  info: (msg, ctx) => console.log(JSON.stringify({ level: 'info', msg, ...ctx })),
  warn: (msg, ctx) => console.warn(JSON.stringify({ level: 'warn', msg, ...ctx })),
  error: (msg, ctx) => console.error(JSON.stringify({ level: 'error', msg, ...ctx })),
};
```
Then in the unlock catch block, call `logger.error(...)` instead of `logger.warn(...)`.

### IN-01: Coverage thresholds below 100% accepted without a forcing-function to close the gap

**File:** `packages/db/vitest.config.ts:16-26`
**Issue:** `functions: 76`, `lines: 96`, `statements: 96` are intentionally below 100% with a comment explaining two unreachable-in-practice paths: `defaultLogger.warn` and the unlock catch block. The comment is accurate, but both gaps are addressable. The unlock catch block (WR-03 above) becomes reachable if the test creates a scenario where the session dies mid-run. `defaultLogger.warn` is reachable by not injecting a logger in the Scenario 3 (orphan-schema) test.

Leaving thresholds low to accommodate untested paths creates ratchet-drift over time — future contributors will assume the bar is 76/96/96, not 100.

**Fix:** After addressing WR-02 and WR-03, revisit coverage. For `defaultLogger.warn`, add one test that passes no logger to a duplicate-object scenario. For the unlock catch block, mock `reserved.unsafe` to throw on the unlock call in an isolated unit test.

### IN-02: `chain` typed as `Record<string, any>` in mock-db — leaks `any` into test assertions

**File:** `packages/api/src/__tests__/helpers/mock-db.ts:4`
**Issue:**
```ts
const chain: Record<string, any> = {};
```
Both `chainable()` and `updateChainable()` use `Record<string, any>` for the internal chain object, and `createMockDb()` returns `any`. This means tests that use the mock lose type checking on mock interactions — a `.returniing()` typo (double-i) would not be caught at compile time.

Per project conventions (`type-safety` rule): "No `any`/`unknown` in interfaces without narrowing." This is a test helper, but it sets the type-safety baseline for all API tests that import from this file.

**Fix:** Define a `ChainableMock` interface or use Vitest's `MockInstance` types. If the fully-typed approach is too verbose, at minimum type `createMockDb()` return as a specific interface so callers get autocomplete on `.select`, `.insert`, etc.
