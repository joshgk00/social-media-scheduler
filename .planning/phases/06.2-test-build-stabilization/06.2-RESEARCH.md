## Summary

This phase splits into two tightly-scoped hardening efforts. Track 1 is pure hygiene: rebuild `@sms/db` + `@sms/shared` dist so downstream package tests stop importing stale compiled code (26 failures), and add a `.returning()` method to the minimal `updateChain` helper in `packages/api/src/__tests__/helpers/mock-db.ts` so commit 08a3e17's `post-lifecycle.service.ts` refactor has a matching test double (4 failures). The worker mock-db already implements a compatible `.returning()` pattern — the planner can use it as the reference shape.

Track 2 is database-reliability work on `packages/db/src/migrate.ts`. The locked CONTEXT.md decisions (D-01, D-04, D-05, D-06) match the fix sketch in 06.1-REVIEW.md §H-01/H-02: session-scoped `pg_advisory_lock` around the whole migration loop; per-migration `client.begin(async tx => ...)` wrapping the statement-apply loop + journal insert; narrowed duplicate-object SQLSTATE set; statement-level duplicate tolerance survives inside the transaction. The one research finding that sharpens the fix: **`max: 1` does not guarantee the same backend PID across calls in postgres-js**; to safely hold a session-scoped advisory lock across multiple queries, the planner should use `sql.reserve()` to pin a single connection, or combine the lock acquisition and `sql.begin()` inside one reserved session. The REVIEW.md sketch is close enough that a careful planner can get it right, but the research flags this explicitly so a task doesn't ship the subtle race.

Track 3 is test coverage for the hardened runner. Real Postgres is required (postgres-js `begin`, advisory locks, and SQLSTATE surfacing are not faithfully emulated by a mock). The existing docker-compose `postgres` service (image `postgres:17-alpine`, exposed to `127.0.0.1:5432` in dev compose) is already paid-for infrastructure. Recommend reusing it via a `DATABASE_URL_TEST` env var with per-test schema isolation rather than adopting `@testcontainers/postgresql` as a new dev dep — Docker is already required to run this project, and testcontainers would add a second container lifecycle to an already-working setup.

**Primary recommendation:**
- Track 1: `pnpm -r build` wired into a root `pretest` script (D-11 option a); add `.returning()` to `updateChain` matching the worker mock's shape.
- Track 2: Use `sql.reserve()` + `sql.begin()` per REVIEW.md fix, lock key `7523098462398n` (a fixed constant documented as frozen forever).
- Track 3: Reuse existing compose `postgres` via `DATABASE_URL_TEST`; per-test schema drop/recreate for isolation; use `Promise.all` with `setTimeout` probes to assert advisory-lock blocking.

## Architecture Patterns

### System Architecture Diagram (hardened migration runner)

```
                                 ┌───────────────────────────────┐
  api/worker process boot ──────▶│ runMigrations(DATABASE_URL)   │
                                 └──────────────┬────────────────┘
                                                │
                                    postgres(url, { max: 1 })
                                                │
                                                ▼
                                 ┌───────────────────────────────┐
                                 │ sql.reserve() → reserved      │
                                 │ (pins ONE backend PID)        │
                                 └──────────────┬────────────────┘
                                                │
                         reserved.unsafe('SELECT pg_advisory_lock(LOCK_KEY)')
                                                │
                                    ┌───────────┴───────────┐
                                    │  lock held (session)  │  second caller blocks here
                                    └───────────┬───────────┘
                                                │
                                    ensureMigrationsTable / readJournal / readAppliedHashes
                                                │
                                                ▼
                                 for each pending migration:
                                    reserved.begin(async tx => {
                                       for each statement:
                                          tx.unsafe(stmt)
                                          catch → if SQLSTATE ∈ DUPLICATE_OBJECT_CODES
                                                     → swallow + log (inside tx)
                                                  else rethrow → tx aborts
                                       tx`INSERT INTO __drizzle_migrations ...`
                                    })
                                                │
                                                ▼
                         finally: reserved.unsafe('SELECT pg_advisory_unlock(LOCK_KEY)')
                                                │
                                                ▼
                         finally: reserved.release(); client.end()
```

Key invariants the test suite must prove:
1. Advisory lock is held across the entire for-each-migration loop.
2. Each migration's statements + journal insert commit as one atomic unit (transaction).
3. A non-duplicate error inside a migration aborts the transaction — no partial row and no journal insert.
4. Duplicate-object errors are swallowed **inside** the transaction (no abort); the transaction still commits.
5. Lock is released even if the transaction aborts (finally path).

### Pattern 1: Advisory-lock + per-migration transaction with postgres-js

**What:** Use `sql.reserve()` to pin a single connection. Acquire the session-scoped advisory lock on the reserved connection. Run each migration inside `reserved.begin(...)` on the same connection. Release lock + connection in `finally`.

**When to use:** Any time a migration needs cross-process serialization AND per-statement atomicity.

**Example (not yet in codebase — target pattern):**

```typescript
// Source pattern: porsager/postgres README §Reserved connections + §Transactions
// Verified: https://github.com/porsager/postgres README
const LOCK_KEY = 7523098462398n; // Frozen forever. Changing this risks double-migration during rolling deploys.

export async function runMigrations(databaseUrl: string, logger: MigrationLogger = defaultLogger) {
  const migrationsFolder = resolve(__dirname, '../drizzle');
  const client = postgres(databaseUrl, { max: 1 });
  try {
    const reserved = await client.reserve();
    try {
      await ensureMigrationsTable(reserved);
      await reserved.unsafe(`SELECT pg_advisory_lock(${LOCK_KEY}::bigint)`);
      try {
        const journal = await readJournal(migrationsFolder);
        const applied = await readAppliedHashes(reserved);

        for (const entry of journal.entries) {
          const sqlPath = join(migrationsFolder, `${entry.tag}.sql`);
          const sql = await readFile(sqlPath, 'utf-8');
          const hash = createHash('sha256').update(sql).digest('hex');
          if (applied.has(hash)) continue;

          await reserved.begin(async (tx) => {
            const statements = splitStatements(sql);
            let skipped = 0;
            for (const statement of statements) {
              try {
                await tx.unsafe(statement);
              } catch (err) {
                const pgCode = (err as { code?: string }).code;
                if (pgCode && DUPLICATE_OBJECT_CODES.has(pgCode)) {
                  skipped += 1;
                  logger.warn('Migration statement skipped — object already present', {
                    migration: entry.tag, pgCode,
                    statementPreview: statement.slice(0, 120).replace(/\s+/g, ' ').trim(),
                  });
                  continue;
                }
                throw err; // Aborts the tx.
              }
            }
            await tx`INSERT INTO "drizzle"."__drizzle_migrations" (hash, created_at) VALUES (${hash}, ${entry.when})`;
            logger.info('Migration applied', { migration: entry.tag, statements: statements.length, skipped });
          });
        }
      } finally {
        await reserved.unsafe(`SELECT pg_advisory_unlock(${LOCK_KEY}::bigint)`);
      }
    } finally {
      await reserved.release();
    }
  } finally {
    await client.end();
  }
}
```

### Pattern 2: Vitest concurrent-caller test for advisory lock

**What:** Two `runMigrations()` calls kicked off with `Promise.all`. Assert that the second call's lock acquisition is delayed by measuring timing OR by checking journal state ordering (first caller writes N journal rows; second caller sees all N applied, writes zero).

**When to use:** Proving the lock actually serializes, not just that it's called.

```typescript
// Source: adapted from BullMQ / connect-redis test patterns + PostgreSQL advisory lock docs
// Verified: https://www.postgresql.org/docs/17/explicit-locking.html §Advisory Locks
it('second concurrent caller waits for first to release the lock', async () => {
  await resetSchemas(testClient);

  const order: string[] = [];
  const first = runMigrations(TEST_URL, makeLogger('first', order));
  // Small delay ensures first starts before second.
  await new Promise((r) => setTimeout(r, 20));
  const second = runMigrations(TEST_URL, makeLogger('second', order));

  await Promise.all([first, second]);

  // First caller applies all 4 migrations; second caller applies zero.
  const [{ count: journalRowCount }] = await testClient`SELECT COUNT(*)::int AS count FROM "drizzle"."__drizzle_migrations"`;
  expect(journalRowCount).toBe(4); // Not 8 — second caller saw them already applied.

  // First caller's "Migration applied" logs precede any of second's log activity.
  const firstAppliedIdx = order.findIndex((e) => e.startsWith('first:applied:0003'));
  const secondStartIdx = order.findIndex((e) => e === 'second:start');
  expect(firstAppliedIdx).toBeLessThan(secondStartIdx);
});
```

The `order` array is the load-bearing assertion — timing alone (e.g., `Date.now()` diffs) is flaky in CI.

### Anti-Patterns to Avoid

- **Holding the lock across `client.end()`.** Session-scoped advisory locks are released when the session ends. If `client.end()` runs before `pg_advisory_unlock`, the lock releases anyway — but the test suite may falsely assert the unlock path ran. Always call `pg_advisory_unlock` explicitly in finally.
- **Using `pg_advisory_xact_lock` instead of `pg_advisory_lock`.** Transaction-scoped locks release at `COMMIT`/`ROLLBACK`. Since this phase's pattern has multiple per-migration transactions inside one lock window, transaction-scoped wouldn't span the whole `runMigrations` call. Use session-scoped per D-01.
- **Using `max: 1` as a substitute for `sql.reserve()`.** `max: 1` serializes queries onto one connection but doesn't guarantee the same backend PID — the driver can drop and reconnect between queries. Session-scoped advisory locks die with the session. [VERIFIED: porsager/postgres README — only `sql.begin()` and `sql.reserve()` guarantee single-connection semantics]
- **Truncating `__drizzle_migrations` between tests without also dropping app tables.** The journal tracking only says what migrations were applied; a stale app schema with extra tables won't be cleaned up by truncating the journal alone. Drop the `drizzle` + app schemas (e.g., `public`) in one shot, then let `ensureMigrationsTable` recreate.
- **Asserting lock-blocked ordering via `Date.now()` comparisons.** CI timing is non-deterministic. Use a shared `order: string[]` array populated by the logger to prove sequence.
- **Rebuilding dist in the test runner itself.** `pretest` runs before `vitest`; calling `tsc` inside a test is a layering violation and creates a retry loop on first run. Keep the build hook at the script level.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-connection session pinning | Custom connection tracker with PID verification | `sql.reserve()` + `.release()` | Native postgres-js API; guarantees same connection. |
| Mutual exclusion across processes | File locks, Redis locks, Postgres row locks | `pg_advisory_lock` | Native Postgres primitive; no new infrastructure; tied to session lifetime. |
| Per-migration atomicity | Manual `BEGIN` / `COMMIT` / `ROLLBACK` via `client.unsafe` | `client.begin(async tx => ...)` | postgres-js's `begin` handles rollback on throw, nested savepoints, proper error propagation. |
| Test-time schema reset | Custom `DROP TABLE` cascade logic per table | `DROP SCHEMA "drizzle" CASCADE; DROP SCHEMA "public" CASCADE; CREATE SCHEMA "public"` | Postgres-native; idempotent; handles any forgotten object. |
| Concurrent test orchestration | Thread-based parallelism | `Promise.all([first, second])` with a shared ordering array | Single-threaded Node is enough to prove serialization; no need for workers. |

**Key insight:** All four migration-runner primitives (connection pinning, mutex, atomicity, schema reset) already exist in postgres-js + Postgres. This phase is entirely "use existing tools correctly" — zero custom infrastructure.

## Common Pitfalls

### Pitfall 1: `max: 1` assumed to pin a single backend PID
**What goes wrong:** Advisory lock is acquired on connection A; next query runs on connection B (driver reconnected between queries); lock isn't visible to connection B; a second concurrent caller can acquire the lock on connection C without waiting.
**Why it happens:** `max: 1` limits pool size but not connection lifetime; the driver is free to close and reopen the single connection for its own reasons (network blips, idle timeouts).
**How to avoid:** Use `sql.reserve()` to explicitly pin one connection until `.release()`. Run all lock/migration queries on the reserved handle.
**Warning signs:** Intermittent test failures on the concurrent-caller scenario; two journal rows with the same hash inserted in quick succession under CI load.

### Pitfall 2: Advisory-lock release in `finally` runs after `client.end()`
**What goes wrong:** `pg_advisory_unlock` call fails with a connection-closed error; log output is noisy; next migration run succeeds anyway (because session-close released the lock) but the error obscures real issues.
**Why it happens:** Nested `finally` blocks run in outside-in order; the outer `client.end()` finally may fire before the inner unlock.
**How to avoid:** Nest the unlock inside the lock's try/finally, which itself is inside the reserve's try/finally, which is inside the client's try/finally. Each layer unwinds in the right order.
**Warning signs:** Migration logs include `connection closed` errors in teardown.

### Pitfall 3: Duplicate-object tolerance swallows a real schema bug inside an aborted transaction
**What goes wrong:** Statement 3 fails with `42710 duplicate_object` (tolerated); statement 4 fails with a real type mismatch; transaction aborts; next boot re-runs all 18 statements; statement 3 tolerated again; statement 4 fails again. Operator sees a crash loop with no indication of the real issue.
**Why it happens:** The test for D-06 must actually trigger a non-duplicate failure AFTER a tolerated duplicate within the same migration, to prove the classifier distinguishes the two.
**How to avoid:** Test scenario explicitly mixes a duplicate-object case with a subsequent real error (e.g., a reference to a non-existent column) in the same migration file. Assert: transaction aborts, journal row NOT inserted, both errors surfaced to the logger.
**Warning signs:** Test only runs pure-duplicate or pure-error migrations; never mixed.

### Pitfall 4: Stale dist guardrail catches the CI case but not the local-dev case
**What goes wrong:** CI passes; a developer rebuilds only `packages/api`, then runs tests and gets stale `@sms/db` / `@sms/shared` imports.
**Why it happens:** A CI-only mtime check in workflow YAML doesn't run on local `pnpm test`.
**How to avoid:** Put the guard in a root `pretest` script that runs everywhere: `"pretest": "pnpm -r build"`. Slower local runs, but zero drift.
**Warning signs:** Test failures reproduce locally but only after a branch switch or a pull.

### Pitfall 5: `reset()` helper in test harness leaves the `drizzle` schema intact
**What goes wrong:** Test A inserts 3 journal rows; test B expects a fresh DB; test B sees 3 pre-applied hashes; test B's "fresh DB applies all migrations" assertion fails.
**Why it happens:** `TRUNCATE __drizzle_migrations` cleans the journal but not app tables; `DROP SCHEMA public CASCADE` without `CREATE SCHEMA public` leaves the test in a broken state; forgetting `"drizzle"` schema leaves the journal.
**How to avoid:** Test helper drops BOTH schemas and recreates `public` in one atomic block. Document the exact order in the helper.
**Warning signs:** Tests pass individually but fail when the suite is run together.

### Pitfall 6: `@sms/shared` dist missed in the `pnpm -r build` order
**What goes wrong:** `@sms/db` depends on `@sms/shared` (workspace:*); if shared dist is absent, db tsc fails. If shared is rebuilt but api's node_modules still link to old dist, api tests fail.
**Why it happens:** `pnpm -r build` respects workspace topology, but if dist is the published artifact (per `main: "dist/index.js"` in both), tsc output must land before any dependent package's test.
**How to avoid:** Root `pretest`: `pnpm -r build` — pnpm builds in topological order by default. Verify with `pnpm -r --filter ...` dry run if unsure.
**Warning signs:** Test run output mentions `Cannot find module '@sms/shared'` or wrong types.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual `client.unsafe('BEGIN')` / `COMMIT` | `client.begin(async tx => ...)` | postgres-js ≥ 3.0 | Automatic rollback on throw; clean error propagation. |
| `max: 1` for session-state preservation | `sql.reserve()` + `.release()` | postgres-js ≥ 3.4 | Explicit connection pinning; correctness over implicit pool behavior. |
| `speakeasy` for TOTP | `otpauth` | 2023+ | (Not this phase, reinforces repo standard.) |
| Ad-hoc schema reset in tests | `DROP SCHEMA CASCADE` + `CREATE SCHEMA` | Standard since Postgres 9.x | One statement, no per-table logic. |

**Deprecated/outdated in scope:**
- **REVIEW.md fix sketch relying on `max: 1` alone.** The sketch works in practice because `max: 1` usually does hold one connection, but research flags it as a latent correctness gap. The planner should upgrade to `sql.reserve()`.

## Security Domain

**Scope:** The migration runner uses `client.unsafe()` to execute arbitrary SQL read from disk. It's security-adjacent (it owns schema consistency) but does not process user input. Limited applicability.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | — |
| V3 Session Management | no | — |
| V4 Access Control | no | — |
| V5 Input Validation | partial | SQL files are code-trusted (checked into `packages/db/drizzle/`); no user input path. But `splitStatements` relies on a literal string delimiter — if a future schema file smuggles the delimiter inside a string literal, statements could split incorrectly. Low-priority (tracked as deferred M-03). |
| V6 Cryptography | no | (SHA-256 hashing is used for journal keys, not security-critical.) |
| V10 Malicious Code | partial | `client.unsafe()` surface reads files from `./drizzle`. Trust boundary: only drizzle-kit writes these. Repo CI should enforce that no hand-edited `*.sql` files land in the drizzle folder outside of a drizzle-kit generate run. |

### Known Threat Patterns for migration-runner + Postgres

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Concurrent migrators race on the journal | Tampering | `pg_advisory_lock` (D-01) |
| Crash-loop on partial migration + duplicate-tolerance masking | Denial of Service | Per-migration `client.begin` transaction (D-04) |
| Unexpected SQLSTATE swallowed as "duplicate" | Tampering | Narrowed `DUPLICATE_OBJECT_CODES` set (D-06) |
| Malicious or malformed `.sql` file | Malicious Code / Injection | File system is trusted; no mitigation in-scope. Flag for future CI enforcement (hash check on drizzle directory). |
| Test database credentials leaked in logs | Information Disclosure | `DATABASE_URL_TEST` is not a secret (points at dev compose); still, tests should not log the URL. |
