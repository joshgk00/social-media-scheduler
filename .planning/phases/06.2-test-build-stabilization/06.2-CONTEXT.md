<decisions>
## Implementation Decisions

### Advisory Lock Semantics (H-01)
- **D-01:** Use `pg_advisory_lock` (blocking wait) for the migration run, matching the fix snippet in `06.1-REVIEW.md`. Success Criterion #4 permits either "waits or exits cleanly" — we choose wait. Rationale: single-host Proxmox deploy means the second caller is almost always a retry of the same container, not a competing migrator; waiting produces a serialized no-op rather than a fail-and-retry loop.
- **D-02:** Use a single stable 64-bit lock key, documented inline with its purpose. Pick one value and freeze it — the key is part of the migration contract.
- **D-03:** Release the lock in a `finally` block so a migration failure still unlocks the key for the next process.

### Per-Migration Atomicity (H-02)
- **D-04:** Wrap each migration's statement loop + journal `INSERT` in a single `client.begin(async tx => ...)` transaction via the `postgres` driver. On failure, the transaction aborts and `__drizzle_migrations` stays unmodified for that migration — the crash-loop scenario from REVIEW.md H-02 becomes impossible.
- **D-05:** Keep duplicate-object tolerance at the **statement level inside the transaction** — per-statement try/catch swallows `DUPLICATE_OBJECT_CODES`, everything else rethrows and aborts the transaction. This preserves drift-tolerance for the orphan-schema baseline case without trading away atomicity.

### Duplicate-Object Codes Scope (M-02 — folded)
- **D-06:** Narrow `DUPLICATE_OBJECT_CODES` to the four SQLSTATE codes that actually arise from drizzle-emitted migration statements: `42P07` (duplicate_table / duplicate_index), `42P06` (duplicate_schema), `42710` (duplicate_object — constraint/type/trigger/extension), `42701` (duplicate_column). Drop `42P04`, `42P03`, `42P05`.

### Migration Test Coverage (M-01)
- **D-07:** Test harness uses a real Postgres instance (not a stub). `postgres` driver behavior around `client.begin`, advisory locks, and SQLSTATE surfacing cannot be faithfully emulated by a mock. Acceptable test approaches in order of preference: (a) `@testcontainers/postgresql` if already a transitive dep; (b) the existing docker-compose `mssql`/`pgsql` containers reused as test fixtures via a `DATABASE_URL_TEST` env var; (c) a lightweight embedded Postgres package. Planner picks one and commits — Claude's Discretion.
- **D-08:** Coverage target: 100% branch coverage on `packages/db/src/migrate.ts`. Required scenarios from the success criteria — fresh DB apply, idempotent re-run, orphan-schema baseline (pre-existing tables tolerated), duplicate-object SQLSTATE tolerance at statement level, real error aborts and rolls back the transaction, concurrent-caller advisory-lock serialization.
- **D-09:** Tests must assert transaction rollback on non-duplicate failure — i.e., verify `__drizzle_migrations` does NOT contain the failed migration's hash after abort, and that partial statements from that migration did NOT persist.

### mock-db `.returning()` Regression
- **D-10:** `mock-db.ts` `updateChain` gains a `.returning()` method that resolves to an array of the SET payload merged over the WHERE-matched rows — i.e., `[{ ...matchedRow, ...setPayload }]`. This matches real `postgres-js` behavior closely enough to satisfy the 4 failing tests introduced by commit `08a3e17`. Same contract applies to `insertChain` if tests need it there; add on first failure.

### Stale Dist Builds — Prevention
- **D-11:** Phase 6.2 builds `@sms/db` and `@sms/shared` dist once as part of execution, but also adds a guardrail so Phase 5's regression does not recur. Options the planner may pick from (Claude's Discretion): (a) `pnpm -r build` wired into a pretest/prebuild script; (b) a CI check that fails if `dist/` mtime is older than any `src/*.ts` mtime; (c) delete checked-in `dist/` entirely and build on install. Pick whichever lands in one commit without changing deploy semantics.

### Review Follow-Ups — In-Scope (folded)
- **D-12:** M-02 (narrow duplicate codes) — folded, see D-06.

### Claude's Discretion
- Specific testcontainer library vs reused docker-compose Postgres for migration tests — planner picks based on existing deps.
- Stale-dist guardrail mechanism — planner picks one of the three D-11 options.
- Whether to break migrate.ts tests across multiple `describe` blocks or one flat file — stylistic.
- Whether `mock-db.ts` `.returning()` lives on `updateChain` only (minimal fix) or extends to `insertChain`/`deleteChain` proactively (broader fix).

<specifics>
## Specific Ideas

- Advisory lock key: single fixed `bigint` constant, defined at top of `migrate.ts` with a comment explaining it's frozen forever and why (any change = operational risk of double-migration during rolling deploys). Planner picks the literal value.
- Tests assert the advisory-lock path by spawning two concurrent `runMigrations()` calls against a shared test database; one should acquire immediately, the second should block until the first releases, then complete as a no-op on the re-read journal.
- M-02 narrowing lands in the same commit as H-01/H-02 so the migrate.ts rewrite is one atomic change.
