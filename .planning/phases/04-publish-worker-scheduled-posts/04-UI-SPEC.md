# Phase 4 — UI Design Contract

> Visual and interaction contract for the Publish Worker & Scheduled Posts phase. Generated by gsd-ui-researcher. Every value is pre-populated from upstream artifacts (CONTEXT.md D-12 through D-26, RESEARCH.md, Phase 3 existing implementation, and `packages/web/src/index.css`). No new design primitives are introduced in this phase — Phase 4 extends the existing Phase 3 `/posts` view, adds modals, and surfaces rate-limit state.

---

## Phase UI Surfaces

This phase touches the following surfaces. Every surface below has a contract in the sections that follow.

| Surface | Scope | Kind |
|---------|-------|------|
| `/posts` list extensions | Error column, Retry kebab action, polling indicator, full-text expand | Extension of Phase 3 page |
| Post History modal | New dialog triggered from kebab, grouped by retry cycle | New component |
| Full Text popover/dialog | New dialog triggered from kebab (SCHED-03 "View full text") | New component |
| Rate limit banner | Inline banner on NewPostPage / EditPostPage when usage ≥ warn threshold | New component |
| Rate limit block error | Inline 409 error surface on post form when budget reached | Extension of existing form error pattern |
| Rate limit settings modal | Edit `monthlyTweetBudget` + `warnThresholdPercent` from profile card | New dialog |
| Rate limit indicator (profile card) | Small usage-vs-budget readout in `/profiles` card | New component |
| `/admin/queues` Bull-Board | Third-party dashboard mounted behind session auth | Embedded third-party panel — not styled from scratch |
| Delete confirmation | Already exists in Phase 3 for posts; Retry does NOT need confirmation (idempotent; safe) | Reuses existing AlertDialog pattern |

## Interaction States (SCHED-01..04, LIMIT-03..04, WORKER-04..07)

### `/posts` list

| State | Behavior |
|-------|----------|
| Loading (first paint) | `Skeleton` rows (already available in `components/ui/skeleton.tsx`), 5 rows, each a single-line skeleton at table row height. No spinner. |
| Polling (subsequent refetches) | Background refetch every 10s via TanStack Query `refetchInterval: 10_000`. Row data updates in place. Polling pauses when `document.hidden` (D-15). A small muted `Updated {N}s ago` indicator sits above the table on the right. |
| Empty (no posts at all) | Phase 3 empty state unchanged. Icon + heading + body + primary CTA `Create Post`. |
| Empty (filtered) | Phase 3 filtered empty state unchanged. |
| Error | Existing Phase 3 error block: `Failed to load posts` heading + helper copy. |
| Row with `failed` status | Row has new error column cell: truncated one-line failure reason + `AlertCircle` icon in destructive color. Hovering the cell shows full text in a `Tooltip`. Row expand (existing Phase 3 pattern) reveals full text + attempt summary teaser. |
| Row with `publishing` status | Disabled kebab actions (matches STATE-02): Edit and Delete are disabled with `Tooltip` explaining `Cannot edit while publishing`. Status badge shows a lucide `Loader2` icon spinning. |
| Row with `scheduled` status | All actions enabled: Edit, Delete, View Full Text, View History. Retry is hidden. |
| Row with `published` status | Retry hidden; Edit hidden; Delete enabled; View History and View Full Text enabled. |
| Row with `draft` status | Retry hidden; View History hidden (no attempts to show). Edit, Delete, View Full Text enabled. |

### History modal

| State | Behavior |
|-------|----------|

| State | Behavior |
|-------|----------|

### Rate limit warning banner

| State | Behavior |
|-------|----------|
| Hidden | `!warnThresholdHit` — banner not rendered |
| Warn (amber) | `warnThresholdHit && !blockThresholdHit` — banner shows; form remains enabled; user can still submit |
| Block (red) | `blockThresholdHit` — banner becomes a destructive block banner; Schedule button is **still clickable** (because the server does the authoritative check on submit) — but on submit the API returns 409, which triggers the block error surface. | Dismissal | **Not dismissible.** Banner remains visible while the condition holds. Re-renders on navigation. |

|-------|----------|

## Accessibility Contract

All Phase 4 UI inherits Phase 2's accessibility baseline (`packages/web/CLAUDE.md`): WCAG AA contrast (4.5:1 text, 3:1 large), `<main>` landmark on every page, semantic HTML before divs, labeled inputs, visible focus rings via `--color-ring`. Phase 4 adds the following requirements:

| Requirement | Detail |
|-------------|--------|
| Dialog focus management | History, Full Text, Rate Limit Settings, and Delete-confirm dialogs must trap focus and restore focus to the trigger element on close — Radix `Dialog` handles this by default. Do not disable it. |
| Dialog labeling | Every `Dialog` uses `DialogTitle` + `DialogDescription` so screen readers announce purpose. No empty titles. |
| Retry button | Must have an accessible name even when rendered as icon + text. The `DropdownMenuItem` text "Retry Post" suffices; do not collapse to icon-only. |
| Status badges | Badge text alone is not sufficient for screen readers when color is the only differentiator between `publishing` and `published`. Badge variants must be paired with their text label, never color-coded without text. |
| Toast notifications | `sonner` must be configured with `aria-live="polite"` (default) — verify by snapshot test. |
| Polling updates | The "Updated {N}s ago" indicator must NOT use `aria-live` — it would announce every 10 seconds and harass assistive tech. Leave as a plain visual element. |
| Rate limit banner | `Alert` component from shadcn is already `role="alert"` — works for screen readers. Do not strip the role. |
| Error column on posts list | Error icon must have `aria-label="Publish failed"` since screen readers otherwise see only the truncated error text. |
| Collapsible retry cycles in History modal | Each `CollapsibleTrigger` must announce expanded/collapsed state (Radix default behavior — don't override). |
| Keyboard navigation | All new kebab menu items must be keyboard-navigable (Radix default). Verified via snapshot test with `userEvent.tab()` in Vitest. |
| Contrast on warning banner | The amber warning color (`yellow-400`) on the dark background must meet WCAG AA. Verify during implementation with a contrast checker. If it fails, darken the banner background via `bg-yellow-400/10 text-yellow-200 border-yellow-400/30` pattern. |
