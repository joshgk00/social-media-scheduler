## Spacing Scale

Tailwind v4 default spacing scale already in use (`p-1` = 4px, `p-2` = 8px, ‚Ä¶). Phase 11 surfaces use ONLY these tokens; no arbitrary `p-[13px]`-style values.

| Token | Value | Usage in Phase 11 |
|-------|-------|-------------------|
| xs | 4px (`p-1`, `gap-1`) | Icon-to-label gap inside `Insert snippet` button; `<mark>` horizontal padding |
| sm | 8px (`p-2`, `gap-2`) | Filter-bar element gaps (Calendar + Snippets page); popover trigger `sideOffset` |
| md | 16px (`p-4`, `gap-4`) | Default popover content padding; calendar event internal padding; dialog body padding-top |
| lg | 24px (`p-6`, `gap-6`) | Page header ‚Üí filter bar ‚Üí grid vertical rhythm on `/calendar` and `/settings/snippets`; dialog default `p-6` |
| xl | 32px (`p-8`) | Empty-state container vertical padding (`py-12` reuse from PostsPage is acceptable ‚Äî that resolves to 48px and is already established in this codebase) |
| 2xl | 48px (`py-12`) | Empty-state vertical padding (matches `PostsPage.tsx` line 498) |
| 3xl | 64px (`py-16`) | Not used in Phase 11 |

**Snippet popover dimensions:** width 320px (`w-80`), max-height 384px (`max-h-96`) so the command list scrolls inside the popover (`<ScrollArea>`-free; `<CommandList>` handles scroll).

**Snippet picker trigger touch target:** `size="sm"` on `<Button>` resolves to 32px height; on `SharedPostFields` mobile width that is acceptable because the textarea sits directly above so a stray miss does not destroy data. No 44px exception required.

**Calendar grid cell padding:** controlled by react-big-calendar; do not override. Inside an event element use `px-2 py-1` (8px / 4px) for text padding.

**Exceptions:** none.

## Typography

Existing Tailwind v4 defaults (in use across the codebase) drive the type scale. Phase 11 declares 4 sizes and 2 weights.

| Role | Size (Tailwind class ‚Üí px) | Weight | Line Height | Usage in Phase 11 |
|------|----------------------------|--------|-------------|-------------------|
| Body | `text-sm` ‚Üí 14px | 400 (regular) | 1.5 (`leading-normal`) | Calendar filter labels, popover items, snippet table cells, search input text, calendar event text, tooltip text |
| Label | `text-xs` ‚Üí 12px | 400 (regular) | 1.5 (`leading-normal`) | Form `FormLabel` (when sized down), helper text under fields, snippet category badge text, "Manage snippets" footer link |
| Heading | `text-lg` ‚Üí 18px | 600 (semibold, `font-semibold`) | 1.2 (`leading-tight`) | Page-level `<h1>` on `/calendar` and `/settings/snippets`; calendar custom-toolbar range label `<h2>`; empty-state heading (matches PostsPage line 509) |
| Display | `text-2xl` ‚Üí 24px | 600 (semibold) | 1.2 (`leading-tight`) | Reserved for any future hero treatment in this phase. Not used by Phase 11 surfaces today; declared so the planner has a clear "do not invent another size" boundary |

Body content (snippet body in `Textarea` + calendar event text > 1 line) uses `leading-normal` (1.5). Headings use `leading-tight` (1.2). Mono font (`font-mono`) is used only inside the snippet body Textarea so users can see whitespace clearly when authoring hashtag sets.

**Two weights only:** 400 (regular) and 600 (semibold). Do not introduce 500 (medium) or 700 (bold) on Phase 11 surfaces.

### Platform-brand colors (NEW ‚Äî required by CAL-02)

The codebase has no existing platform color tokens (verified via `rg`). Phase 11 introduces three semantic tokens added to `packages/web/src/index.css` `@theme` block. Each is tested for ‚â• 3:1 contrast against `--color-background` (dark zinc) ‚Äî they're used as 4-pixel left borders and 12% background tints, never as foreground text, so the AA 4.5:1 rule does not apply, but 3:1 against the surface is required for the indicator to read.

| Token | OKLCH | Hex equivalent | Used for |
|-------|-------|----------------|----------|
| `--color-platform-twitter` | `oklch(0.700 0.165 235)` | ~`#1DA1F2` (Twitter sky-blue) | Calendar event left-border + 12% background tint when `platform === 'twitter'` |
| `--color-platform-linkedin` | `oklch(0.450 0.130 245)` | ~`#0A66C2` (LinkedIn deeper blue) | Calendar event left-border + 12% background tint when `platform === 'linkedin'` |
| `--color-platform-facebook` | `oklch(0.520 0.180 265)` | ~`#1877F2` (Facebook blue-violet) | Calendar event left-border + 12% background tint when `platform === 'facebook'` |

Tailwind v4 with CSS variables auto-exposes these as `bg-platform-twitter`, `border-platform-twitter`, etc. Executors apply via `eventPropGetter` returning `border-l-4 border-platform-{p} bg-platform-{p}/10` based on event payload. **Conflict events override** the left-border to `border-l-destructive` per D-13 ‚Äî destructive (red) trumps brand color so users always notice conflicts.

### Accent reserved for

The 10% accent (`bg-primary`) is reserved for ONLY these elements on Phase 11 surfaces:

1. The "Save snippet" / "Create snippet" submit button in the snippet dialog (form primary CTA)
2. The "Manage snippets" footer link inside the picker popover, when in `:hover` / `:focus-visible` (it's `text-muted-foreground` at rest; underlines on hover)
3. The "Today" button in the calendar custom toolbar
4. The active state of the `Tabs` segments (M / W / D switcher and Scheduled / Queued / Both toggle) ‚Äî shadcn `Tabs` already styles the active segment with `bg-background text-foreground` over a `bg-muted` track, which is the established accent treatment

Accent is NOT used on filter chips, NOT on the `Insert snippet` button (that one is `variant="outline"` to keep visual weight low next to the textarea), NOT on calendar events, NOT on table rows.

### Destructive reserved for

`destructive` color is used on these Phase 11 elements only:

1. The "Delete snippet" confirm button in the snippet delete dialog (matches existing `ConfirmDestructiveDialog`)
2. The 4-pixel left border on calendar entries flagged `hasConflict: true` (CAL-04 indicator, D-13)
3. Form-field error message text under invalid snippet name / body inputs (RHF + shadcn `FormMessage` default ‚Äî already destructive)

## Interaction & Keyboard Contracts

| Surface | Keyboard / interaction contract |
|---------|----------------------------------|
| Snippet picker trigger | Activates on `Enter` or `Space`; opens popover; focus moves to `<CommandInput>` automatically (shadcn `Popover` + `Command` default) |
| Snippet picker popover | `ArrowDown` / `ArrowUp` move highlight; `Home` / `End` jump first / last; `Enter` inserts highlighted snippet at textarea cursor and closes popover; `Esc` closes without inserting and returns focus to the trigger button (D-04) |
| Snippet picker search filter | Case-insensitive substring match on snippet `name` only (D-04 ‚Äî no fuzzy match in Phase 11) |
| Snippet insertion behavior | Inserts `body` at the textarea's `selectionStart..selectionEnd`; if the user has a range selected, that range is replaced; caret lands immediately AFTER the inserted text; textarea regains focus (D-02) |
| Snippets page table rows | Each row is a `<tr>`; row actions live in a `<DropdownMenu>` triggered by an icon-only `<Button variant="ghost" size="icon">` with `aria-label="Snippet actions for {name}"` |
| Search input on three views | Debounced 250ms (Claude's-discretion default per CONTEXT); URL state via `setSearchParams(..., { replace: true })` so back-button skips keystroke history; clearing the input restores the unfiltered list within one debounce cycle |
| Calendar view switcher (Tabs) | `ArrowLeft` / `ArrowRight` move between M / W / D (shadcn `Tabs` default); switching re-queries the windowed API |
| Calendar prev / today / next | Three buttons, `aria-label="Previous {view}"`, `aria-label="Today"`, `aria-label="Next {view}"`; `Today` resets focus to today's cell |
| Calendar event click | Navigates to `/posts/:id/edit` (CAL-02); `Enter` on a focused event does the same |
| Calendar empty-slot click | Navigates to `/posts/new?scheduledAt=ISO` where `ISO` is the clicked datetime in the user's IANA timezone (CAL-02). On Day view, an empty cell is a 30-minute slot; on Week view, a cell is one hour; on Month view, an empty day cell pre-fills date only with no time component |
| Calendar conflict tooltip | Shows on hover AND on keyboard focus of the event (`focus-visible`); content matches `ScheduleConflictBanner` wording: "Another post on this profile is scheduled within 5 minutes of this time: "{textPreview}" at {date}." |

**Accessibility notes (binding):**

- Every Phase 11 page wraps content in `<main>` per `packages/web/CLAUDE.md`.
- Calendar route `<header>`, `<section aria-label="Calendar filters">`, and the calendar grid live inside `<main>`.
- All inputs have associated `<label>` (RHF + shadcn `<FormLabel>` handles this for the snippet form).
- The `<mark>` element used for `ts_headline` rendering is announced by screen readers as "marked text" ‚Äî that is the desired behavior, no extra aria needed.
- Tooltip content for conflict indicators MUST also be reachable via the event element's `aria-describedby` so screen-reader users hear the conflict text without needing to hover.
