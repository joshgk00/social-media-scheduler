    2. Apply the migration via the project's migration runner. Two equivalent commands; prefer the second (the project's hardened runner from Phase 6.2 — advisory-locked, per-migration transactional):
       - Option A (drizzle-kit, less safe — does not use the project's hardened runner): `pnpm --filter @sms/db exec drizzle-kit migrate`
       - Option B (preferred — invokes `packages/db/src/migrate.ts` runMigrations()): `pnpm --filter @sms/db exec tsx src/migrate.ts`
       Use Option B. If `tsx` is not installed at the workspace root, fall back to Option A — the runner will still apply the migration through drizzle-kit, just without the advisory lock.

    3. The migration runner will:
       - Acquire `pg_try_advisory_lock` (per Phase 6.2 H-01)
       - Read `__drizzle_migrations` to find pending entries
       - Run `0009_phase-11-snippets-fts-calendar.sql` inside a single transaction (per Phase 6.2 H-02)
       - Tolerate duplicate-object SQLSTATE codes if re-run

       Expected output: a log line indicating `0009_phase-11-snippets-fts-calendar` was applied successfully. If you see `generation expression is not immutable` — the regconfig literal `'english'` is missing in the search_vector DDL; fix plan 03's SQL and re-run.

    4. **Verify** the live schema. Run the following commands inside the running pgsql container (the docker-compose pgsql service uses postgres user / phase2password!; database name comes from `.env` `DATABASE_URL`):

       Check 1 — snippets table present:
           docker exec pgsql psql -U postgres -d $DB_NAME -c "\d snippets"
       Expected: shows columns `id`, `user_id`, `name`, `category`, `body`, `created_at`, `updated_at`; foreign key to `users(id) ON DELETE CASCADE`; unique index `snippets_user_lower_name_unq`.

       Check 2 — posts.search_vector is GENERATED:
           docker exec pgsql psql -U postgres -d $DB_NAME -c "\d posts"
       Expected: line includes `search_vector | tsvector | | | generated always as (to_tsvector('english'::regconfig, ((text || ' '::text) || COALESCE(notes, ''::text)))) stored`. Also `tag_search_vector | tsvector` (no GENERATED clause).

       Check 3 — GIN index exists:
           docker exec pgsql psql -U postgres -d $DB_NAME -c "\di posts_fts_idx"
       Expected: one row showing `posts | posts_fts_idx | index | gin`.

       Check 4 — trigger exists:
           docker exec pgsql psql -U postgres -d $DB_NAME -c "\dft post_tags_after_change"
       OR (more direct):
           docker exec pgsql psql -U postgres -d $DB_NAME -c "SELECT tgname, tgenabled FROM pg_trigger WHERE tgname = 'post_tags_after_change';"
       Expected: one row with `tgenabled = O`.

       Check 5 — query planner uses the GIN index:
           docker exec pgsql psql -U postgres -d $DB_NAME -c "EXPLAIN ANALYZE SELECT id, ts_rank(search_vector || tag_search_vector, plainto_tsquery('english', 'sample')) AS rank FROM posts WHERE (search_vector || tag_search_vector) @@ plainto_tsquery('english', 'sample') ORDER BY rank DESC LIMIT 10;"
       Expected: `Bitmap Index Scan on posts_fts_idx` appears in the plan. (If the planner falls back to Seq Scan because the table is small, force `SET enable_seqscan = off;` for the EXPLAIN — that is acceptable; the index existing and being usable is the requirement, the planner's choice on a tiny dev dataset is not.)

       If any check fails, the migration is incomplete — investigate, fix the migration SQL in plan 03, drop the partially-applied state (`DROP TABLE snippets`, `ALTER TABLE posts DROP COLUMN search_vector`, `ALTER TABLE posts DROP COLUMN tag_search_vector`, `DROP TRIGGER post_tags_after_change ON post_tags`, `DROP FUNCTION post_tags_refresh_vector`, `DROP FUNCTION refresh_post_tag_vector`, `DELETE FROM __drizzle_migrations WHERE hash = ...`), and re-run.

    5. Capture the verification command outputs to the plan SUMMARY (paste excerpts of `\d posts`, `\di posts_fts_idx`, and EXPLAIN ANALYZE) so future audits can confirm the migration shipped correctly.
