## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (snippet CRUD) | Same as plan 05; client-side validation is convenience, server is authoritative |
| client snippet body → DOM | Snippet body inserted into a `<textarea>` `value`, never `innerHTML` — no XSS surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-09-01 | Tampering (XSS via snippet body) | SnippetPicker insert | mitigate | Body inserted into a textarea via the controlled `value` prop only — never `innerHTML` / `dangerouslySetInnerHTML`; React escapes by default |
| T-11-09-02 | Logic flaw (cursor lands at position 0) | onPointerDown capture | mitigate | Pitfall 4: capture selectionStart/End in onPointerDown BEFORE focus moves; Test A asserts mid-text insertion preserves both sides |
| T-11-09-03 | Information Disclosure (caching another user's snippet list) | useSnippets queryKey | accept | Single-user app; queryKey `['snippets']` is per browser session; no shared cache risk |
| T-11-09-04 | Tampering (race between optimistic update and server response) | useCreateSnippet | mitigate | Use TanStack Query `onSuccess: invalidate` (no optimistic update); avoids stale state on duplicate-name 409 |
| T-11-09-05 | Information Disclosure (snippet body in console.log) | dev components | mitigate | Per CLAUDE.md §Module Structure / Logging: no console.log of snippet body. The cmdk component renders body content but only inside the controlled DOM, never logs it |
</threat_model>
