# Phase 06.1: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 5
**Status:** findings

## Summary

Phase 6.1 is narrowly scoped production wiring plus one deviation (a hand-rolled migration runner in `packages/db/src/migrate.ts`). The three Plan 01/02 files (`packages/api/src/index.ts`, `docker-compose.yml`, `nginx/nginx.conf`, `Dockerfile`) execute their plans faithfully — small, defensible changes, no secret exposure, existing shutdown and proxy patterns preserved.

The migration runner deviation is the highest-risk change in the phase. It replaces drizzle's transactional `migrate()` with a best-effort per-statement runner that swallows duplicate-object SQLSTATE codes. That solves the "prior `drizzle-kit push` left orphan tables" operational pain point, but it introduces two real risks worth addressing before this pattern bakes into long-term operations: (1) no mutual-exclusion lock, so two API containers starting simultaneously will race on the journal; and (2) each migration is no longer atomic — a failure mid-migration leaves the DB partially migrated with no journal row, so the next boot re-runs the partial set and the duplicate-object tolerance papers over the partial state rather than reporting it. Neither is blocking for a single-host Proxmox deploy today, but both should be tracked as known limitations.

The phase is also missing tests for the new migration runner. Given its security-adjacent responsibility (schema consistency and the `client.unsafe()` surface) this is a medium-severity gap per repo standards.

## High

### H-01: `migrate.ts` — no advisory lock; concurrent migrators race on the journal

**File:** `packages/db/src/migrate.ts:69-92`
**Issue:** The runner reads `readAppliedHashes()`, then iterates and applies one statement at a time, then inserts the hash row. If two API replicas boot simultaneously (or the API and a separate migration job run in parallel), both see the same "not yet applied" set and both attempt to apply the same migration. `ALTER TABLE ... DROP NOT NULL` and `INSERT INTO __drizzle_migrations` are not mutually exclusive between connections, so the journal can end up with duplicate hash rows and the "duplicate object" tolerance masks the race instead of preventing it. drizzle's built-in `migrate()` wraps the whole thing in a single transaction per migration which gives row-level locking via the `INSERT ... RETURNING` pattern; this implementation has neither a transaction nor a lock.

**Fix:** Take a Postgres advisory lock for the duration of the migration run. Single-connection pool (`max: 1`) already isolates within one process; advisory lock extends that across processes.

```ts
export async function runMigrations(databaseUrl: string, logger: MigrationLogger = defaultLogger) {
  const migrationsFolder = resolve(__dirname, '../drizzle');
  const client = postgres(databaseUrl, { max: 1 });
  try {
    await ensureMigrationsTable(client);
    // Any stable int64 — drizzle uses 0xdcbf35ea9c2d5f3a in some variants; pick one and document it.
    const LOCK_KEY = 7523098462398n;
    await client.unsafe(`SELECT pg_advisory_lock(${LOCK_KEY}::bigint)`);
    try {
      const journal = await readJournal(migrationsFolder);
      const applied = await readAppliedHashes(client);
      for (const entry of journal.entries) {
        // ... existing body
      }
    } finally {
      await client.unsafe(`SELECT pg_advisory_unlock(${LOCK_KEY}::bigint)`);
    }
  } finally {
    await client.end();
  }
}
```

With a single-replica Proxmox deploy this is latent; add it now to prevent a nasty, hard-to-reproduce corruption the first time someone scales the API service or runs `docker compose up` while an existing container is still doing its migration pass on boot.

### H-02: `migrate.ts` — migrations no longer atomic; partial apply leaves DB + journal out of sync

**File:** `packages/db/src/migrate.ts:128-160`
**Issue:** The per-statement loop does not run inside a transaction. If statement 10 of 18 fails with a non-duplicate error (say, a foreign-key violation because a referenced table doesn't exist yet because a prior duplicate-tolerated statement was supposed to create it but a subtle `IF NOT EXISTS` mismatch left it absent), statements 1–9 stay applied but `INSERT INTO __drizzle_migrations` never runs. Next boot, the runner re-executes statements 1–9 — all duplicate-tolerated — then re-hits statement 10 and fails again. The operator sees a crash loop with no indication that the DB is in a partial state.

drizzle's built-in `migrate()` wraps each migration in a transaction; that's why it aborts on `CREATE TABLE "queues"` in the scenario that motivated this replacement. The fix from the plan's perspective was correct (tolerate duplicates) but throwing out the transaction wrapper discarded the atomicity guarantee.

**Fix:** Wrap the per-migration statement loop + journal insert in a transaction. `postgres` supports this via `client.begin`:

```ts
await client.begin(async (tx) => {
  for (const statement of statements) {
    try { await tx.unsafe(statement); }
    catch (err) {
      const pgCode = (err as { code?: string }).code;
      if (pgCode && DUPLICATE_OBJECT_CODES.has(pgCode)) { skipped += 1; continue; }
      throw err;
    }
  }
  await tx`INSERT INTO "drizzle"."__drizzle_migrations" (hash, created_at) VALUES (${hash}, ${entry.when})`;
});
```

This preserves the duplicate-tolerance that motivated the rewrite while restoring drizzle's "all-or-nothing per migration file" guarantee. Combined with H-01's advisory lock, the runner matches drizzle's operational contract.

## Medium

### M-01: `migrate.ts` — no tests for a file that now owns schema consistency

**File:** `packages/db/src/migrate.ts` (entire file)
**Issue:** Repo CLAUDE.md mandates "Security-critical code ... 100% branch coverage" and "test both success AND failure paths." This runner has several branches that aren't exercised: duplicate-object codes hit/miss, non-duplicate error rethrow, journal hash match (skip) vs miss (apply), empty statement filter, malformed journal. It also has a `client.unsafe()` surface that executes arbitrary SQL read off disk — the only thing stopping injection is trust in the `drizzle/` directory, which the test suite should pin. Given this file replaces a well-tested library function, the absence of any unit tests is a regression in confidence.

**Fix:** Add `packages/db/src/__tests__/migrate.test.ts` with at least: (a) fresh DB applies all migrations and journal rows appear; (b) re-run is a no-op; (c) drifted schema (manually create the `queues` table first) is tolerated with warn logs; (d) a genuinely malformed SQL statement (e.g., `SELECT FROM WHERE;`) rethrows instead of being swallowed; (e) journal with a missing `.sql` file raises a clear error.

### M-02: `migrate.ts` — duplicate-object codes list is over-broad

**File:** `packages/db/src/migrate.ts:23-32`
**Issue:** The set includes `42P04 duplicate_database` and `42P03 duplicate_cursor` and `42P05 duplicate_prepared_statement`. None of those can arise from a drizzle schema migration: migrations don't `CREATE DATABASE`, don't declare cursors, and don't use prepared statements by name. Including them in the tolerance set widens the "swallow" net without a real use case. The risk is low but the point of fail-fast is to surface the unexpected, and a duplicate-database error — if it ever came back from a future schema change — would almost certainly indicate operator error worth aborting on.

**Fix:** Narrow the set to the four codes that actually arise from `CREATE TABLE`, `CREATE TYPE`, `CREATE INDEX`, `ADD COLUMN`, `ADD CONSTRAINT`, and `CREATE SCHEMA`:

```ts
const DUPLICATE_OBJECT_CODES = new Set([
  '42P07', // duplicate_table (CREATE TABLE, CREATE INDEX)
  '42P06', // duplicate_schema
  '42710', // duplicate_object (CONSTRAINT, TYPE, TRIGGER, EXTENSION)
  '42701', // duplicate_column
]);
```

### M-03: `migrate.ts` — `splitStatements` uses a brittle string delimiter

**File:** `packages/db/src/migrate.ts:162-167`
**Issue:** The splitter looks for the literal string `--> statement-breakpoint`. drizzle emits that marker reliably today, but the runner is now coupled to drizzle's internal output format without a version check or a test. If drizzle-kit changes the breakpoint text in a future release (including casing or whitespace), the runner will silently treat the whole migration as one big statement. Most statements would still work (postgres.js `.unsafe()` supports multi-statement strings), but the per-statement duplicate tolerance collapses — one duplicate column in statement 3 will abort statements 4–18.

**Fix:** Pin the drizzle version in `packages/db/package.json` to a narrow range (repo standard: tilde `~0.45.x`), add a runtime check that at least one breakpoint is present for migrations with > 1 statement, or read the `breakpoints: true` flag from the journal entry and fall back to a single-statement apply when false. Minimum bar: add a test that pins the expected delimiter so a silent drizzle bump breaks CI.

### M-04: `index.ts` — migrations run on the default Postgres connection, not the app pool, but share `DATABASE_URL`

**File:** `packages/api/src/index.ts:27,30`
**Issue:** Line 27 runs `runMigrations(DATABASE_URL)` which opens its own `postgres(url, { max: 1 })` client. Line 30 then calls `createDbClient(DATABASE_URL)` which opens the app pool. Both clients hold open connections simultaneously until migrations finish and `client.end()` is called in the `finally` block. On a cold Postgres with `max_connections = 100` defaults this is fine. If the operator later tunes Postgres down or adds a connection pooler, the migration connection competes with the app pool. This is low-impact today but worth flagging because the "migrate then bind" sequencing is intentional and the two clients could share a URL pool if migrate() accepted an already-open `Sql`.

**Fix:** Consider threading the already-constructed `pgClient` into `runMigrations` so there's exactly one connection in use during boot. Out of scope for this phase; track as a followup.

## Low

### L-01: `index.ts` — `parseInt` on PORT has no validation

**File:** `packages/api/src/index.ts:64`
**Issue:** `parseInt(process.env.PORT || '3000', 10)` returns `NaN` if `PORT` is a non-numeric string (e.g., `PORT=web`). `app.listen(NaN, ...)` will then throw asynchronously and the process will exit via the `main().catch()` handler, but the error message will be opaque. This is a pre-existing pattern, not introduced in 6.1, so it's low priority — flagging because it sits two lines above new code.

**Fix:** Validate: `const port = Number.parseInt(process.env.PORT ?? '3000', 10); if (!Number.isFinite(port) || port < 1 || port > 65535) throw new Error(\`Invalid PORT: ${process.env.PORT}\`);`

### L-02: `nginx.conf` — `log_format` emits `$request` including query string; correlation-ID header name is custom

**File:** `nginx/nginx.conf:21-25`
**Issue:** `$request` in the access log includes the full request URI, which means query strings land in logs. For this app most query strings are harmless (filters, pagination), but login/reset flows sometimes carry tokens in `?token=...` style URLs. This is a latent PII/secret-log risk rather than a bug introduced by 6.1. Plan 02 explicitly did not touch headers/logging, so this is a pre-existing concern worth filing for a future infra cleanup.

**Fix:** If any route ever takes a secret in a query parameter, switch to `$uri` + filtered `$args` or add an `if ($arg_token) { set $args ""; }` sanitizer. For now, audit existing routes to confirm no query-string secrets exist.

### L-03: `Dockerfile` — `web-production` stage reinstalls wget on every rebuild

**File:** `Dockerfile:64-68`
**Issue:** `RUN apk add --no-cache wget` is required by the compose healthcheck but executes on every rebuild that invalidates the `nginx:1.27-alpine` base layer. Minor. The layer is cached under normal operation.

**Fix:** None required — the `apk add` is correctly cached by Docker. Flagging only because an alternative would be to use `curl` (also missing from the alpine base) or switch the healthcheck to nginx's built-in `/stub_status`. Current approach is the simplest and matches Plan 02 Task 2's intent.

## Info

### I-01: `index.ts` — `dotenv` side effect at module scope violates repo "no top-level side effects" rule (pre-existing)

**File:** `packages/api/src/index.ts:1-7`
**Issue:** Root CLAUDE.md says "Env vars read inside functions at runtime, never at module scope" and "factory functions — no top-level side effects." Lines 1–7 load `.env` at import time, and lines 19–23 call `requireEnv()` at module scope. The API entry point genuinely is a process entry (not a library), so this is defensible, but it means this file can't be imported by tests without triggering dotenv load and env validation. This is pre-existing, not a 6.1 regression.

**Fix:** None for this phase — the pattern is established across all three entry points (api, worker, scripts). If a future refactor extracts `main()` into a factory, move these calls inside it.

### I-02: `docker-compose.yml` — no explicit port binding on `nginx` to `127.0.0.1`

**File:** `docker-compose.yml:72-73`
**Issue:** Port maps to `${NGINX_PORT:-8080}:80`. Repo CLAUDE.md says "Dev ports: `127.0.0.1` only, never `0.0.0.0`." The 6.1 research notes this rule applies to dev compose, not prod. This file is the production compose and is intentionally publicly bindable. No fix needed — logging for clarity during audit.

**Fix:** None. Production binding to `0.0.0.0` is intentional and documented.

### I-03: `migrate.ts` — console.log default logger bypasses pino

**File:** `packages/db/src/migrate.ts:53-56`
**Issue:** `defaultLogger` uses `console.log` / `console.warn` so when called without an explicit logger the output is plain JSON to stdout but not correlated with the pino-based API logger used elsewhere. In practice `packages/api/src/index.ts:27` calls `runMigrations(DATABASE_URL)` without passing a logger, so the migration output uses this default. pino's JSON shape has additional fields (pid, hostname, time ISO format) that aren't present here. Minor cosmetic / observability gap.

**Fix:** Either pass the API's pino logger explicitly (`runMigrations(DATABASE_URL, pinoAdapter)`), or change the default to use a minimal pino child logger. Track as a small followup; not blocking.
