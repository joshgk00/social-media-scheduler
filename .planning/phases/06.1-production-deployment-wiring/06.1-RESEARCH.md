# Phase 6.1: Production Deployment Wiring - Research

**Researched:** 2026-04-16
**Domain:** Docker Compose production wiring, Express route mounting, nginx reverse proxy, Vite production builds
**Confidence:** HIGH

## Summary

Phase 6.1 is a gap closure phase addressing four specific defects found during the v1.0 milestone audit. The issues are: (1) the API entry point never instantiates a storage backend or transcode queue, so the media router's conditional guard in `app.ts` evaluates to false and all `/api/media/*` routes are dead; (2) `SESSION_SECRET` is missing from the production `docker-compose.yml` environment block, crashing the API on startup; (3) no web production Dockerfile target exists and the `web_dist` volume is never populated, so nginx serves an empty directory; (4) five Phase 4 requirements (SCHED-01, SCHED-02, SCHED-03, LIMIT-04, LIMIT-05) are verified as implemented but missing from SUMMARY frontmatter.

During research, a fifth defect was discovered: the production nginx config uses `proxy_pass http://api_backend/;` (with trailing slash), which strips the `/api/` prefix from forwarded requests. Since all Express route handlers define their paths with the `/api/` prefix (e.g., `router.post('/api/posts', ...)`), no API route would match through nginx in production. This must be fixed alongside the other four issues. [VERIFIED: codebase grep of nginx/nginx.conf line 45 and packages/api/src/routes/*.ts]

**Primary recommendation:** Wire the four audit gaps plus the nginx proxy_pass fix. All changes are small, isolated, and require no new dependencies. Total scope is approximately 5 files modified and 2-3 Dockerfile lines added.

### Pattern 2: nginx proxy_pass URI Behavior

**What:** The trailing slash on `proxy_pass` controls whether the location prefix is stripped.

**Current state (broken):**
```nginx
location /api/ {
    proxy_pass http://api_backend/;  # <-- trailing slash strips /api/ prefix
}
```

With this config, `GET /api/posts` arrives at Express as `GET /posts`. But Express routes are defined as `router.post('/api/posts', ...)`, so nothing matches. [VERIFIED: nginx/nginx.conf line 45, packages/api/src/routes/posts.ts, auth.ts, profiles.ts, queues.ts]

**Required state:**
```nginx
location /api/ {
    proxy_pass http://api_backend;  # <-- no trailing slash preserves /api/ prefix
}
```

With no trailing slash, `GET /api/posts` arrives at Express as `GET /api/posts`, matching correctly. [CITED: nginx proxy_pass documentation -- http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_pass]

### Pattern 3: Multi-stage Docker Build for Web Assets

**What:** The existing Dockerfile `build` stage already runs `pnpm -r build`, which builds the web package (producing `packages/web/dist/`). A new `web-production` target needs to extract those assets into a location accessible by nginx.

Two approaches considered:

**Approach A: Dedicated nginx Dockerfile stage (RECOMMENDED)**
Add a Dockerfile stage that copies the built web assets from the `build` stage into the nginx image. This replaces the named `web_dist` volume with a custom nginx image containing the assets. This is the standard Docker pattern for SPA serving.

```dockerfile
# Web + nginx production image
FROM nginx:1.27-alpine AS web-production
COPY nginx/nginx.conf /etc/nginx/nginx.conf
COPY --from=build /app/packages/web/dist /usr/share/nginx/html
```

The `docker-compose.yml` nginx service would then use `build: { context: ., target: web-production }` instead of `image: nginx:1.27-alpine` + `web_dist` volume. This eliminates the empty volume problem entirely. [ASSUMED]

**Approach B: Init container populates volume**
Keep the named volume, add a `web-build` service that runs once to copy files into it. More complex and fragile (ordering issues, stale data on rebuild). Not recommended.

### Anti-Patterns to Avoid

- **Do not add `MEDIA_STORAGE_BACKEND` or `MEDIA_DIR` to docker-compose.yml** -- both have correct defaults (`'local'` and `'./data/media'`), and the `media_data` volume is already mounted at `/app/data/media`. Adding them would be unnecessary configuration surface. [VERIFIED: packages/shared/src/storage/index.ts defaults]
- **Do not change Express route paths** -- all routers use `/api/` prefix consistently. Fix the nginx config instead of rewriting dozens of route definitions.
- **Do not create a separate `web` Docker service for production** -- the dev compose has a web service (Vite dev server). Production serves static files from nginx directly. No runtime Node process needed.

## Common Pitfalls

### Pitfall 1: nginx proxy_pass Trailing Slash
**What goes wrong:** All API requests through nginx return 404 because the `/api/` prefix is stripped.
**Why it happens:** nginx `proxy_pass http://backend/;` with trailing slash replaces the matched location prefix with the proxy_pass URI path. Since Express routes include `/api/` in their definitions, the stripped requests don't match.
**How to avoid:** Remove the trailing slash: `proxy_pass http://api_backend;`
**Warning signs:** Health endpoint works (has its own `location = /health` block) but all other API routes fail through nginx.

### Pitfall 2: Forgetting transcodeQueue Shutdown
**What goes wrong:** The transcode BullMQ queue is created in `index.ts` but not closed during SIGTERM, leaking a Redis connection.
**Why it happens:** The shutdown handler was written before the transcode queue existed.
**How to avoid:** Add `transcodeQueue.close()` to the shutdown sequence, matching the pattern for `publishQueueService.publishQueue.close()` and `notificationQueue.close()`.
**Warning signs:** Redis connection warnings on container restart.

### Pitfall 3: Docker Build Cache Invalidation
**What goes wrong:** After changing the Dockerfile to add a `web-production` stage, `docker compose build` may use cached layers and not rebuild the web assets.
**Why it happens:** Docker caches build stages. If source files changed but the `COPY . .` instruction's context hash hasn't been recalculated, stale assets may be served.
**How to avoid:** Use `docker compose build --no-cache` or `docker compose up --build` after Dockerfile changes. In steady-state, the `COPY . .` step correctly invalidates when source changes.

### Pitfall 4: Static Asset Cache Headers Missing
**What goes wrong:** Browsers don't cache Vite-hashed assets, causing unnecessary network requests.
**Why it happens:** Production nginx.conf has gzip enabled but no `Cache-Control` headers for static assets.
**How to avoid:** Add `location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$` block with `Cache-Control: public, max-age=31536000, immutable` for hashed assets. Vite includes content hashes in filenames, making immutable caching safe. CLAUDE.md requires: "Production nginx: gzip + static asset cache headers".
**Warning signs:** Large repeat page loads in network inspector.

### Pitfall 5: web_dist Volume Collision
**What goes wrong:** If switching from volume-based to COPY-based static serving, the old `web_dist` named volume definition in docker-compose.yml becomes unused and should be removed.
**Why it happens:** Leftover configuration from the original design.
**How to avoid:** Remove the `web_dist` volume from both the nginx service volumes list and the top-level `volumes:` section when switching to the built nginx image approach.

## Open Questions (RESOLVED)

1. **nginx Non-Root User on Port 80**
   - What we know: CLAUDE.md requires `USER` directive for non-root containers. nginx typically needs root to bind port 80.
   - What's unclear: Whether nginx:1.27-alpine supports non-root binding on port 80.
   - RESOLVED: Skip the `USER` directive in the `web-production` Dockerfile stage. nginx:1.27-alpine manages its own user (binds port 80 as root, then drops to the `nginx` user via its default config). Explicit `USER appuser` breaks port-80 bind. This is a documented exception to the repo-wide non-root rule and is captured in the Plan 02 Task 2 action and threat model (T-06.1-04).

2. **Phase 4 SUMMARY Format for Requirements**
   - What we know: The 04-04-SUMMARY.md has a `requirements:` field in frontmatter listing SCHED-01 etc., but the audit tool did not recognize it. The 04-05-SUMMARY.md has no requirements field at all.
   - What's unclear: Exactly what frontmatter format the audit tool expects for "claiming" a requirement.
   - RESOLVED: Use a `requirements_satisfied:` YAML list in the SUMMARY frontmatter (distinct from the existing `requirements:` / `tags:` keys). This is the key the audit tool reads. Applied in Plan 03 Task 2.
