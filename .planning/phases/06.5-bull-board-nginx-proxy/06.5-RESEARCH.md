# Phase 6.5: Nginx Proxy Completion - Research

**Researched:** 2026-04-22
**Domain:** nginx reverse proxy configuration, Docker Compose volume mounts, static file serving
**Confidence:** HIGH

## Summary

Phase 6.5 closes two production-blocking integration gaps identified by the v1.0 re-audit (2026-04-21). Both gaps have the same root cause: the nginx.conf SPA catch-all (`try_files $uri $uri/ /index.html`) intercepts requests to `/media/` and `/admin/` paths before they can reach the Express API server.

The Express API already serves media files via `express.static` at `/media` (app.ts:91) and mounts Bull-Board at `/admin/queues` (app.ts:67-72). Both work when hitting api:3000 directly. The problem is purely at the nginx layer -- two `location` blocks are missing, and the nginx container (`web-production` Dockerfile target) does not mount the `media_data` Docker volume. The fix is configuration-only: edit nginx.conf, nginx.dev.conf, and docker-compose.yml. No application code changes are needed.

The scope is deliberately narrow. Three files modified, zero new dependencies, zero application logic changes.

**Primary recommendation:** Add `location /media/` and `location /admin/` proxy blocks to both nginx.conf and nginx.dev.conf, and mount the `media_data` volume on the nginx container in docker-compose.yml. Verify with curl commands through the nginx port.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Media file serving | CDN / Static (nginx) | API / Backend (Express) | nginx proxies `/media/` to Express which serves via `express.static`. nginx must forward the request, not serve files itself -- Express owns the static mount and the media_data volume. |
| Bull-Board dashboard | API / Backend (Express) | CDN / Static (nginx) | Bull-Board is an Express middleware mounted at `/admin/queues`. nginx must proxy to Express, not attempt to serve it as static content. |
| Docker volume access | Infrastructure (Docker Compose) | -- | The `media_data` volume must be mounted on the nginx container OR (preferred) nginx must proxy to the API container which already has the volume mounted. |

**Key architectural decision: proxy vs. direct serve.** There are two ways nginx could serve `/media/` files:

1. **Proxy to Express (RECOMMENDED):** nginx forwards `/media/` to `api_backend`, Express serves via `express.static`. This is simpler because the API container already mounts `media_data` at `/app/data/media` and Express is already configured to serve it. No volume mount needed on nginx.

2. **Direct serve from nginx:** Mount `media_data` on the nginx container and use `root` or `alias` directives. This bypasses Express but requires duplicate volume mounts and MIME type configuration. It also bypasses any future auth checks on media URLs.

Option 1 is correct here. The Express `express.static` call at app.ts:91 already handles MIME types, and future phases may add auth gating to media files. Proxying preserves this flexibility. [VERIFIED: app.ts:90-91 `express.static(mediaDir)`]

**However:** For the `/media/` proxy, there is a subtlety. The `docker-compose.yml` nginx container does NOT need the `media_data` volume if we proxy to Express. The API container already has it. The audit report's success criterion #2 ("docker-compose.yml nginx service mounts the media_data volume so express.static can serve files") is describing a direct-serve approach. Since we're proxying to Express instead, the volume mount on nginx is unnecessary -- Express is the one running `express.static`, and it already has the volume. The success criterion is still met in spirit: uploaded media files are served through the published nginx port, which is the actual goal.

### Pattern 1: nginx Location Block Ordering

**What:** nginx evaluates location blocks in a specific priority order. The SPA catch-all (`location /`) has the lowest priority (prefix match). Adding more specific `location /media/` and `location /admin/` prefix matches will take priority over the catch-all. [CITED: http://nginx.org/en/docs/http/ngx_http_core_module.html#location]

**When to use:** Whenever adding new proxy paths to an nginx config that has a catch-all SPA block.

**Current nginx.conf location blocks (in file order):**
1. `location = /health` -- exact match (highest priority)
2. `location /api/` -- prefix match
3. `location /assets/` -- prefix match with cache headers
4. `location /` -- prefix match (catch-all, lowest priority)

**After Phase 6.5:**
1. `location = /health` -- exact match (highest priority)
2. `location /api/` -- prefix match
3. `location /media/` -- prefix match [NEW]
4. `location /admin/` -- prefix match [NEW]
5. `location /assets/` -- prefix match with cache headers
6. `location /` -- prefix match (catch-all, lowest priority)

nginx prefix matching selects the longest matching prefix, so `/media/` and `/admin/` will always win over `/` for their respective paths. File ordering does not matter for prefix matches, but placing them before the catch-all improves readability. [CITED: http://nginx.org/en/docs/http/ngx_http_core_module.html#location]

### Pattern 2: proxy_pass Without Trailing Slash

**What:** `proxy_pass http://api_backend;` (no trailing slash) preserves the original request URI. `proxy_pass http://api_backend/;` (with trailing slash) replaces the matched location prefix. [CITED: http://nginx.org/en/docs/http/ngx_http_proxy_module.html#proxy_pass]

**For `/media/`:** Express serves at `app.use('/media', express.static(mediaDir))`. A request for `/media/profiles/1/2026/04/abc.jpg` must arrive at Express as `/media/profiles/1/2026/04/abc.jpg`. Use `proxy_pass http://api_backend;` without trailing slash.

**For `/admin/`:** Bull-Board base path is set to `/admin/queues` in admin.ts:32. A request for `/admin/queues` must arrive at Express as `/admin/queues`. Use `proxy_pass http://api_backend;` without trailing slash. [VERIFIED: packages/api/src/routes/admin.ts:32 `serverAdapter.setBasePath('/admin/queues')`]

### Pattern 3: Dev vs. Prod Config Parity

**What:** The project maintains separate nginx configs: `nginx.conf` (production) and `nginx.dev.conf` (development). Both need the same proxy blocks for `/media/` and `/admin/`.

**Key difference between prod and dev configs:**

| Aspect | nginx.conf (prod) | nginx.dev.conf (dev) |
|--------|-------------------|---------------------|
| SPA serving | `try_files` from `/usr/share/nginx/html` (baked into image) | `proxy_pass http://web:5173` (Vite dev server) |
| `/api/` proxy | `proxy_pass http://api_backend` | `proxy_pass http://api_backend/` (note: trailing slash -- strips `/api/` prefix in dev) |
| Asset caching | `location /assets/` with immutable headers | Not present (Vite handles HMR) |

**Important:** The dev config at line 45 has `proxy_pass http://api_backend/;` with a trailing slash, which strips the `/api/` prefix. This is the OPPOSITE behavior from prod. The `/media/` and `/admin/` blocks should use NO trailing slash in BOTH configs, because Express expects the full path for both endpoints. [VERIFIED: nginx/nginx.dev.conf:45 vs nginx/nginx.conf:45]

### Anti-Patterns to Avoid

- **Serving media directly from nginx with `alias` or `root`:** Bypasses Express, requires duplicate volume mount, loses future auth gating capability, and duplicates MIME type configuration.
- **Adding auth_basic to nginx for /admin/:** Bull-Board is already auth-gated by the `requireAuth` Express middleware (admin.ts:43). Adding nginx-level auth would create double authentication.
- **Using regex locations for simple prefix paths:** `location /media/` (prefix) is sufficient. No need for `location ~ ^/media/(.*)$` which adds regex overhead and complexity.
- **Adding the media_data volume to the nginx container:** Unnecessary if proxying to Express. Express already has the volume. Adding it to nginx creates a second mount point that serves no purpose and widens the attack surface.

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: SPA Catch-All Swallowing API Paths
**What goes wrong:** Requests to `/media/image.jpg` return the React SPA HTML instead of the image file. The browser renders a broken image because it received HTML with content-type text/html.
**Why it happens:** nginx `location /` with `try_files $uri $uri/ /index.html` matches everything. Without a more specific `location /media/` block, media requests fall through to the catch-all.
**How to avoid:** Add explicit `location /media/` and `location /admin/` blocks BEFORE the catch-all in nginx.conf. nginx longest-prefix matching ensures they take priority.
**Warning signs:** Images show as broken in the browser. Inspecting the network tab shows 200 OK with content-type text/html for image URLs. [VERIFIED: this is the exact bug described in the audit report]

### Pitfall 2: proxy_pass Trailing Slash Mismatch
**What goes wrong:** Express receives the wrong URL path and returns 404.
**Why it happens:** A trailing slash in `proxy_pass http://api_backend/;` strips the matched location prefix. `/media/image.jpg` becomes `/image.jpg` at Express, which doesn't match `app.use('/media', ...)`.
**How to avoid:** Use `proxy_pass http://api_backend;` without trailing slash for `/media/` and `/admin/` blocks.
**Warning signs:** nginx returns 404 for media URLs that work when hitting api:3000 directly.

### Pitfall 3: Forgetting the Dev Config
**What goes wrong:** Media and admin work in production but not in development.
**Why it happens:** Only `nginx.conf` is updated, but development uses `nginx.dev.conf`.
**How to avoid:** Update both configs in lockstep. Both need `/media/` and `/admin/` location blocks.
**Warning signs:** Works in production Docker Compose but fails with `docker compose -f docker-compose.yml -f docker-compose.dev.yml up`.

### Pitfall 4: Bull-Board WebSocket/SSE Upgrade Headers
**What goes wrong:** Bull-Board's real-time job updates don't work through nginx.
**Why it happens:** Bull-Board may use polling or SSE for live updates. The proxy block needs HTTP/1.1 and Upgrade headers if WebSocket is used.
**How to avoid:** Include the same proxy headers as the `/api/` block: `proxy_http_version 1.1`, `Upgrade`, `Connection` headers. These are harmless if unused and prevent issues if Bull-Board uses WebSocket.
**Warning signs:** Bull-Board dashboard loads but job status doesn't update in real time.

### Pitfall 5: nginx Log Query String PII Exposure (Tech Debt L-02)
**What goes wrong:** Media URLs may contain user-identifiable path components (profile IDs). nginx default log format includes the full request URI including path.
**Why it happens:** The existing `log_format main` includes `$request` which logs the full URL path.
**How to avoid:** This is existing tech debt (L-02 from Phase 6.1 audit), not a new issue. Note it but don't fix it in this phase -- it's out of scope.
**Warning signs:** Profile IDs visible in nginx access logs.
