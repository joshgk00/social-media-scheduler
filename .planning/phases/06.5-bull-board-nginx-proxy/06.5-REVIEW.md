# Phase 6.5: Code Review Report

**Reviewed:** 2026-04-22T09:30:00Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Reviewed two nginx configuration files that add `/media/` and `/admin/` location blocks to both the production and development reverse proxy configs. These blocks fix a routing gap where the catch-all location (`/` serving SPA or proxying to Vite) was swallowing requests meant for the Express API's static media serving and Bull-Board admin dashboard.

The changes are structurally correct. Both new blocks use the right `proxy_pass` form (no trailing slash) so Express receives the full URI path, matching the server-side mount points at `/media` (static) and `/admin/queues` (Bull-Board). The `/admin/` block includes WebSocket upgrade headers and the `/media/` block omits them, which is appropriate for their respective use cases. Placement before the catch-all block is correct for nginx longest-prefix matching.

No critical issues or warnings found. Two informational observations below.

## Info

### IN-01: Proxy header duplication across location blocks

**File:** `nginx/nginx.conf:57-77` and `nginx/nginx.dev.conf:57-77`
**Issue:** Four location blocks in each file repeat the same five-line proxy header set (`Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Request-ID`). This is not a bug but increases maintenance surface -- adding or changing a header requires edits in four places per file. Nginx supports extracting common directives into a separate file via `include`.
**Fix:** Optional cleanup for a future pass -- extract shared proxy headers into a snippet file:
```nginx
# /etc/nginx/proxy-headers.conf
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
proxy_set_header X-Request-ID $request_id;
```
Then `include /etc/nginx/proxy-headers.conf;` inside each location block. Not blocking -- the current inline approach is common and works fine.

### IN-02: No explicit proxy timeouts on /admin/ location block

**File:** `nginx/nginx.conf:66-77` and `nginx/nginx.dev.conf:66-77`
**Issue:** The `/admin/` block includes WebSocket upgrade headers, indicating Bull-Board may hold long-lived connections (SSE for real-time job status). Without explicit `proxy_read_timeout` or `proxy_send_timeout` directives, these default to 60 seconds. If Bull-Board uses SSE for live updates, the connection will be dropped after 60s of inactivity. This may cause the dashboard to show stale data until manually refreshed.
**Fix:** If Bull-Board uses SSE or WebSocket for live updates, consider adding explicit timeouts:
```nginx
location /admin/ {
    # ... existing directives ...
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
}
```
Not blocking for initial deployment -- the dashboard remains fully functional with manual refresh. Can be tuned after observing actual behavior.
