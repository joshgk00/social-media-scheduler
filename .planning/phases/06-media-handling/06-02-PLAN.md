<objective>
Build the media upload API: multer middleware, image processing service (thumbnail generation + validation), video upload handling (enqueue transcode job), media status polling, soft-delete, and association endpoints. The upload decouples from post save -- files upload independently and get associated to a post when saved.

Purpose: This provides the server-side media pipeline that the frontend drop zone (Plan 04) uploads to, and that the transcode worker (Plan 03) processes.
Output: POST /api/media/upload, GET /api/media/:id/status, DELETE /api/media/:id endpoints, media.service.ts with image/video processing logic.

    **1. Multer middleware** (`packages/api/src/middleware/media-upload.ts`):
    - Configure multer with disk storage:
      - `destination`: `os.tmpdir()` (temp directory; files move to StorageBackend after processing)
      - `filename`: `${randomUUID()}-${Date.now()}${path.extname(file.originalname)}` to avoid collisions
    - Set limits: `fileSize: 100 * 1024 * 1024` (100MB max -- largest allowed per Facebook video D-03; server validates per-platform after)
    - Set `fileFilter`: reject any MIME type not in the union of all `PLATFORM_MEDIA_LIMITS` allowedImageTypes and allowedVideoTypes. This is a first-pass filter; per-platform validation happens in the service.
    - Export `mediaUpload` as the configured multer instance with `.single('file')`

    **2. Media service** (`packages/api/src/services/media.service.ts`):

    `processImageUpload(params: { tempFilePath: string, originalName: string, mimeType: string, profileId: string, storage: StorageBackend, db: Db })`:
    - Use `sharp(tempFilePath).metadata()` to extract width, height, format
    - Generate storage key: `media/${profileId}/${year}/${month.padStart(2,'0')}/${randomUUID()}.${formatToExt(format)}` where formatToExt maps sharp format names to extensions (jpeg->jpg, png->png, gif->gif, webp->webp)
    - Generate thumbnail key: same path but `${uuid}_thumb.${ext}`
    - Read original with `sharp(tempFilePath).rotate().toBuffer()` (EXIF rotation fix), save via `storage.save(key, buffer, mimeType)`
    - Generate thumbnail: `sharp(tempFilePath).rotate().resize(300, undefined, { withoutEnlargement: true }).toBuffer()`, save via `storage.save(thumbnailKey, thumbBuffer, mimeType)`
    - Insert `post_media` row: `{ postId: null, filePath: key, fileName: originalName, mimeType, fileSize: buffer.length, width, height, thumbnailPath: thumbnailKey, sortOrder: 0, transcodeStatus: 'not_applicable' }`
    - Clean up temp file with `fs.promises.unlink(tempFilePath)` in finally block
    - Return `{ id, fileName, mimeType, fileSize, thumbnailUrl: storage.getUrl(thumbnailKey), transcodeStatus: 'not_applicable' }`

    `processVideoUpload(params: { tempFilePath: string, originalName: string, mimeType: string, fileSize: number, profileId: string, storage: StorageBackend, db: Db, transcodeQueue: Queue })`:
    - Generate storage key: `media/${profileId}/${year}/${month}/${randomUUID()}_original${path.extname(originalName)}`
    - Save original to storage via `storage.save(key, fs.createReadStream(tempFilePath), mimeType)`
    - Insert `post_media` row: `{ postId: null, filePath: key, fileName: originalName, mimeType, fileSize, thumbnailPath: null, sortOrder: 0, transcodeStatus: 'pending' }`
    - Enqueue BullMQ job: `transcodeQueue.add(JOB_NAMES.transcodeVideo, { mediaId: row.id, inputKey: key, profileId }, { jobId: 'transcode-' + row.id, attempts: 1 })`
    - Clean up temp file in finally block
    - Return `{ id: row.id, fileName: originalName, mimeType, fileSize, thumbnailUrl: null, transcodeStatus: 'pending' }`

    `getMediaStatus(db: Db, mediaId: string)`:
    - Query `post_media` by id, return `{ id, transcodeStatus, transcodeError }`

    `softDeleteMedia(db: Db, storage: StorageBackend, mediaId: string)`:
    - Update `post_media` set `deletedAt = new Date()` where id = mediaId and deletedAt is null
    - Do NOT delete the file from storage here (cleanup job handles permanent deletion per D-13/D-14)

    `associateMediaToPost(db: Db, postId: string, mediaIds: string[])`:
    - For each mediaId at index i: update `post_media` set `postId = postId, sortOrder = i` where `id = mediaId AND postId IS NULL`
    - This prevents double-claiming (RESEARCH.md Pitfall 5)

    **3. Media routes** (`packages/api/src/routes/media.ts`):
    Follow `createXxxRouter({ db })` factory pattern from existing routes.

    `createMediaRouter({ db, storage, transcodeQueue })` returns Express Router:

    `POST /upload`:
    - Middleware: `requireAuth`, `mediaUpload.single('file')`
    - Extract `profileId` from `req.body.profileId` (required, validates as UUID)
    - Extract `platform` from `req.body.platform` (required, one of 'twitter', 'linkedin', 'facebook')
    - Validate file size against `PLATFORM_MEDIA_LIMITS[platform]` -- if image and exceeds maxImageSizeMb, return 400 `{filename} exceeds the {size} MB limit.`; same for video and maxVideoSizeMb
    - Validate MIME type against platform's allowedImageTypes/allowedVideoTypes -- if not allowed, return 400 `{filename} is not a supported file type.`
    - If image MIME: call `processImageUpload`, return 201 with result
    - If video MIME: call `processVideoUpload`, return 201 with result

    `GET /:id/status`:
    - Middleware: `requireAuth`
    - Call `getMediaStatus(db, req.params.id)`, return 200

    `DELETE /:id`:
    - Middleware: `requireAuth`
    - Call `softDeleteMedia(db, storage, req.params.id)`, return 204

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client upload -> multer | Untrusted file content from browser; validated by multer filter then sharp/service |
| File path generation | UUID-based keys prevent path traversal; original filenames stored as metadata only |
| Platform validation | profileId and platform from request body must be validated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Tampering | File upload content | mitigate | Multer MIME filter rejects non-media types; sharp metadata() validates image files are real images (not renamed executables); video files validated by ffmpeg during transcode |
| T-06-05 | Denial of Service | Oversized uploads | mitigate | Multer fileSize limit 100MB (absolute max); per-platform limits enforced in route handler before processing; rejects with 400 and clear message |
| T-06-06 | Tampering | Path traversal via filename | mitigate | Original filename stored as metadata only; storage key is UUID-generated; StorageBackend has path traversal guard |
| T-06-07 | Elevation of Privilege | Unauthenticated upload | mitigate | All media routes behind `requireAuth` middleware; CSRF token required on POST/DELETE |
| T-06-08 | Information Disclosure | Media URL guessability | accept | UUIDs in paths provide sufficient unpredictability; single-user app; no sensitive media expected |
</threat_model>
