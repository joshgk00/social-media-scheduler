## Shared Patterns

### Transaction-Internal Association

**Source:** `packages/api/src/services/post.service.ts` lines 80-106 (create) and 135-245 (update)
**Apply to:** All mediaIds handling in `createPost` and `updatePost`

Both create and update wrap their mutations in `db.transaction(async (tx) => { ... })`. All association calls (tags, media) must use `tx`, not `db`. The transaction mock in tests uses `fn(db)` to simulate this.

### Guard-Then-Call Pattern

**Source:** `packages/api/src/services/post.service.ts` lines 93-103 (tag guard)
**Apply to:** Media association calls

```typescript
// Tags: guard on length, then call
if (tagIds.length > 0) { /* ... */ }

// Media: guard on defined AND length, then call
if (input.mediaIds && input.mediaIds.length > 0) { /* ... */ }
```

The extra `input.mediaIds` truthy check is needed because `mediaIds` is optional (unlike `tagIds` which defaults to `[]` at line 78).

### Dissociate-Before-Reassociate on Update

**Source:** `packages/api/src/services/post.service.ts` lines 231-244 (tag dissociate pattern)
**Apply to:** Media update path

Tags use `DELETE` (since `postTags` is a join table); media uses `UPDATE SET postId = null` (since `postMedia` rows contain file data that outlives the association).

### Error Class

**Source:** `packages/api/src/services/post.service.ts` lines 49-55
**Apply to:** No new error classes needed -- existing `PostServiceError` covers any validation errors from media association
