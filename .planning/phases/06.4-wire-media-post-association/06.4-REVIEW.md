# Phase 06.4: Code Review Report

**Reviewed:** 2026-04-18
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

The media-post association wiring in `post.service.ts` is structurally sound — `associateMediaToPost` and `softDeleteMediaForPost` are called in the right places and the new tests cover the key call-site contracts. However, there are four warnings that affect correctness or reliability:

1. `getPostById` never fetches associated media, so `createPost` and `updatePost` return responses that silently omit media even after a successful write.
2. `softDeleteMediaForPost` always returns 0 because `.returning()` is absent from its Drizzle update chain.
3. `deletePost` runs the soft-delete and the hard-delete as separate, non-transactional operations — a failure between them leaves media marked deleted while the post survives.
4. `associateMediaToPost` wraps its updates in a redundant inner transaction when called with a transaction handle from the post service, creating unnecessary savepoints.

---

## Warnings

### WR-01: `getPostById` does not fetch media — `createPost` and `updatePost` return responses without a `media` field

**File:** `packages/api/src/services/post.service.ts:316`

**Issue:** `getPostById` is the shared read-path used as the return value for both `createPost` (line 116) and `updatePost` (line 266). It fetches the post row and joins `postTags`/`tags`, but never queries `postMedia`. Any media successfully associated in those write paths is invisible to the API caller. Clients that rely on the returned post object to confirm media attachment (e.g., for thumbnail display) will see no media.

**Fix:** Add a `postMedia` join inside `getPostById`, mirroring the existing `postTags` join:

```typescript
const mediaRows = await db
  .select({
    id: postMedia.id,
    fileName: postMedia.fileName,
    mimeType: postMedia.mimeType,
    fileSize: postMedia.fileSize,
    thumbnailPath: postMedia.thumbnailPath,
    sortOrder: postMedia.sortOrder,
    transcodeStatus: postMedia.transcodeStatus,
  })
  .from(postMedia)
  .where(and(eq(postMedia.postId, post.id), isNull(postMedia.deletedAt)))
  .orderBy(postMedia.sortOrder);

return { ...post, tags: postTagRows, media: mediaRows };
```

### WR-02: `softDeleteMediaForPost` always returns 0 — missing `.returning()` on the Drizzle update chain

**File:** `packages/api/src/services/media.service.ts:224`

**Issue:** The function does `Array.isArray(updatedRows) ? updatedRows.length : 0` to count soft-deleted rows, but without `.returning()` the Drizzle update call resolves to a non-array query result object. `Array.isArray(updatedRows)` is always `false`, so the count is always 0. The conditional log in `deletePost` (line 279) that reads `if (softDeletedMediaCount > 0)` therefore never fires, even when media files were marked for cleanup.

**Fix:** Add `.returning({ id: postMedia.id })` to get back the affected rows:

```typescript
const updatedRows = await db
  .update(postMedia)
  .set({ deletedAt: new Date() })
  .where(and(eq(postMedia.postId, postId), isNull(postMedia.deletedAt)))
  .returning({ id: postMedia.id });

return updatedRows.length;
```

### WR-03: `deletePost` runs soft-delete and hard-delete without a wrapping transaction — partial failure leaves media orphaned

**File:** `packages/api/src/services/post.service.ts:269`

**Issue:** `softDeleteMediaForPost` (line 277) and `db.delete(posts)` (line 282) are two separate database operations with no transaction around them. If `softDeleteMediaForPost` succeeds but `db.delete` fails or skips (e.g., the post is in a non-deletable state, or there is a transient DB error), the post survives but its media rows have `deletedAt` set. Those files will be irreversibly swept by the 30-day cleanup pipeline even though the post still exists and is valid.

**Fix:** Wrap both operations in a transaction, and move the status check inside it:

```typescript
export async function deletePost(db: Db, userId: string, postId: string): Promise<boolean> {
  return db.transaction(async (tx) => {
    const softDeletedMediaCount = await softDeleteMediaForPost(tx, postId);
    if (softDeletedMediaCount > 0) {
      logger.info({ postId, softDeletedMediaCount }, 'Soft-deleted media for post deletion');
    }

    const deletedRows = await tx.delete(posts)
      .where(and(
        eq(posts.id, postId),
        eq(posts.userId, userId),
        inArray(posts.status, [...DELETABLE_STATES]),
      ))
      .returning({ id: posts.id });

    if (deletedRows.length === 0) {
      // Transaction will roll back the soft-delete if we throw here
      const existingPost = await tx
        .select({ id: posts.id, status: posts.status })
        .from(posts)
        .where(and(eq(posts.id, postId), eq(posts.userId, userId)));

      if (existingPost.length === 0) {
        throw new PostServiceError('Post not found', 404);
      }
      throw new PostServiceError('This post cannot be deleted in its current state.', 409);
    }

    logger.info({ postId, userId }, 'Post deleted');
    return true;
  });
}
```

### WR-04: `associateMediaToPost` opens a redundant inner transaction when called with a `tx` handle

**File:** `packages/api/src/services/media.service.ts:284`

**Issue:** `associateMediaToPost` always calls `db.transaction(async (tx) => { ... })` internally. Both callers in `post.service.ts` pass a transaction handle (`tx`) rather than the root `db` instance (lines 108 and 259). In Drizzle's postgres-js adapter, calling `.transaction()` on an already-open transaction handle creates a PostgreSQL savepoint. This works correctly, but the savepoint is unnecessary — the caller's transaction already provides rollback safety. It also means every call allocates a savepoint and requires a matching release, adding latency for no benefit.

**Fix:** Remove the inner `db.transaction()` and execute the updates directly. If standalone use (outside a transaction) is needed, document that callers are responsible for providing the correct context:

```typescript
export async function associateMediaToPost(
  db: Db,
  postId: string,
  mediaIds: string[],
): Promise<void> {
  for (let i = 0; i < mediaIds.length; i++) {
    await db
      .update(postMedia)
      .set({ postId, sortOrder: i })
      .where(and(eq(postMedia.id, mediaIds[i]), isNull(postMedia.postId)));
  }
}
```

## Info

### IN-01: No test coverage for `getPostById` — the media-return gap from WR-01 will not be caught by CI

**File:** `packages/api/src/__tests__/services/post.test.ts`

**Issue:** There is no `describe('getPostById', ...)` block in the unit test file. Given that WR-01 describes a real behavior gap (media rows never returned), there is no test that would fail today to signal it. Once WR-01 is fixed, a dedicated test for `getPostById` should verify that both tags and media are returned.

**Fix:** Add a `getPostById` describe block covering at minimum: post not found → `null`, post found → tags populated, post found → media populated (sorted by `sortOrder`, excluding soft-deleted rows).

### IN-02: `createPostUpdateMockDb` `selectCallIndex` is never reset between calls — silent mock exhaustion

**File:** `packages/api/src/__tests__/services/post.test.ts:109`

**Issue:** `selectCallIndex` is a closure variable that increments with every `.select()` call and is never reset. If a test triggers more select calls than the mock anticipates, all subsequent calls silently return `makeSelectChain([])`. This means a test that runs more DB reads than expected passes quietly instead of failing loudly.

**Fix:** The existing pattern is acceptable since each test creates a fresh mock via `createPostUpdateMockDb()`. Document this assumption with a comment, or add an assertion at the test level if over-calling is suspected in a given test case.

---

### IN-03: `checkConflicts` time window is inclusive on both endpoints — semantics worth documenting

**File:** `packages/api/src/services/post.service.ts:434`

**Issue:** `gte(posts.scheduledAt, windowStart)` and `lte(posts.scheduledAt, windowEnd)` mean a post scheduled exactly at `targetTime ± 5 minutes` is flagged. This is intentional (conservative conflict detection) but the inclusive-boundary choice is not documented in comments or the PRD reference, so future maintainers may change it unintentionally.

**Fix:** Add a short inline comment: `// ±5 min window, inclusive on both ends — intentionally conservative`.
