# Phase 6.4: Wire Media-Post Association - Research

**Researched:** 2026-04-17
**Domain:** Integration wiring -- connecting existing media service to post routes
**Confidence:** HIGH

## Summary

This is a narrow integration gap closure. All the building blocks already exist: `associateMediaToPost()` in `media.service.ts` is exported and tested, the Zod schemas (`createPostSchema` / `updatePostSchema`) already include `mediaIds?: string[]`, and the frontend already sends `mediaIds` in POST/PATCH request bodies. The publish worker's media-readiness gate in `post-lifecycle.service.ts:173-190` correctly queries `post_media WHERE post_id = ? AND transcode_status IN ('pending','processing')`.

The gap: `CreatePostInput` and `UpdatePostInput` interfaces in `post.service.ts` do not include `mediaIds`. The `createPost()` and `updatePost()` functions never receive or forward `mediaIds`. The route handlers in `routes/posts.ts` never call `associateMediaToPost()`. Result: `post_media.post_id` stays NULL after post creation, the publish worker's COUNT query always returns 0, and MEDIA-05 is dead code.

**Primary recommendation:** Add `mediaIds?: string[]` to the service-layer interfaces, call `associateMediaToPost()` inside the existing transactions in `createPost()` and `updatePost()`, and add tests verifying the end-to-end flow from route through service to database.

### Pattern 1: Association Inside Existing Transaction

**What:** Call `associateMediaToPost` inside the same transaction that creates/updates the post, using the transaction handle (`tx`) instead of `db`.

**When to use:** Always -- atomic consistency between post creation and media association prevents orphaned state.

**Example:**
```typescript
// In post.service.ts createPost()
const post = await db.transaction(async (tx) => {
  const [insertedPost] = await tx.insert(posts).values({...}).returning();

  if (tagIds.length > 0) { /* existing tag logic */ }

  // Associate media after post insert, inside same transaction
  if (input.mediaIds && input.mediaIds.length > 0) {
    await associateMediaToPost(tx, insertedPost.id, input.mediaIds);
  }

  return insertedPost;
});
```

[VERIFIED: codebase grep -- associateMediaToPost accepts Db type parameter which is compatible with transaction handle]

### Pattern 2: Update Path -- Dissociate Then Re-associate

**What:** On update, the caller may change which media items are attached. Current `associateMediaToPost` only sets `postId` on rows where `postId IS NULL`. For updates, the route should first clear existing associations, then re-associate.

**When to use:** PATCH `/api/posts/:id` when `mediaIds` is provided.

**Example:**
```typescript
// In post.service.ts updatePost(), inside the transaction:
if (input.mediaIds !== undefined) {
  // Clear existing media associations for this post
  await tx
    .update(postMedia)
    .set({ postId: null })
    .where(eq(postMedia.postId, postId));

  // Re-associate with new list
  if (input.mediaIds.length > 0) {
    await associateMediaToPost(tx, postId, input.mediaIds);
  }
}
```

[ASSUMED] -- This dissociate-then-reassociate pattern is the simplest approach. An alternative is diff-based (only update changed rows), but for small media counts (max 4 images on Twitter, max 10 on Facebook) the simple approach is correct and fast.

### Anti-Patterns to Avoid

- **Calling associateMediaToPost outside the transaction:** Would create a window where the post exists but media is not attached, or where the post creation fails but media is orphaned with a stale postId.
- **Silently dropping invalid mediaIds:** If a user sends a mediaId that does not exist or already belongs to another post, the `WHERE postId IS NULL` clause in `associateMediaToPost` silently skips it. Consider whether to validate upfront and return an error.

## Common Pitfalls

### Pitfall 1: Transaction Scope Mismatch

**What goes wrong:** Calling `associateMediaToPost(db, ...)` instead of `associateMediaToPost(tx, ...)` inside the transaction. The association runs outside the transaction boundary, breaking atomicity.
**Why it happens:** `associateMediaToPost` accepts a `Db` type, which is the same type for both the main client and a transaction handle in Drizzle.
**How to avoid:** Always pass `tx` (the transaction handle) when calling inside a transaction block.
**Warning signs:** Media associations persist even when post creation rolls back.

### Pitfall 2: Update Path Doesn't Clear Old Associations

**What goes wrong:** On PATCH, only calling `associateMediaToPost` without first clearing existing associations. If the user removes a media item, it stays attached.
**Why it happens:** `associateMediaToPost` only sets `postId` on rows where `postId IS NULL` -- it doesn't unset existing associations.
**How to avoid:** Clear existing associations before re-associating on update.
**Warning signs:** Removed media items still appear attached to the post.

### Pitfall 3: mediaIds Validation Gap

**What goes wrong:** User sends a mediaId for a media item that belongs to another post or doesn't exist. `associateMediaToPost` silently skips it (WHERE postId IS NULL), and the user doesn't know their media wasn't attached.
**Why it happens:** No validation that the mediaIds actually reference existing, unassigned media rows.
**How to avoid:** Optionally validate that all mediaIds exist and have `postId IS NULL` before associating. Return a 400 if any are invalid.
**Warning signs:** Posts appear to have fewer media items than the user intended.

### Pitfall 4: Circular Import

**What goes wrong:** `post.service.ts` already imports `softDeleteMediaForPost` from `media.service.ts`. Adding `associateMediaToPost` to that import is safe (same module). But if `media.service.ts` ever imports from `post.service.ts`, a circular dependency forms.
**Why it happens:** Two services that reference each other's functions.
**How to avoid:** Keep the dependency one-directional: post.service -> media.service, never the reverse.
**Warning signs:** Already safe -- `media.service.ts` has no imports from `post.service.ts`. [VERIFIED: codebase grep]

## Code Examples

### Current associateMediaToPost implementation (no changes needed)

```typescript
// Source: packages/api/src/services/media.service.ts:284-297
export async function associateMediaToPost(
  db: Db,
  postId: string,
  mediaIds: string[],
): Promise<void> {
  await db.transaction(async (tx) => {
    for (let i = 0; i < mediaIds.length; i++) {
      await tx
        .update(postMedia)
        .set({ postId, sortOrder: i })
        .where(and(eq(postMedia.id, mediaIds[i]), isNull(postMedia.postId)));
    }
  });
}
```

**Note:** `associateMediaToPost` creates its own transaction internally. When called from within `createPost`'s transaction, the nested `db.transaction()` call will use the already-active transaction handle if `db` is actually a `tx`. In Drizzle, passing a transaction handle to `db.transaction()` reuses the existing transaction (nested transactions become savepoints). This is correct behavior. ## Open Questions

1. **Nested transaction behavior in Drizzle**
   - What we know: `associateMediaToPost` wraps its logic in `db.transaction()`. When called from `createPost`'s transaction with `tx` as the `db` argument, it should reuse the outer transaction.
   - What's unclear: Whether Drizzle's postgres-js adapter supports nested transactions as savepoints or creates a new connection.
   - Recommendation: Either (a) verify with a quick test, or (b) refactor `associateMediaToPost` to accept a `tx` directly without wrapping in its own transaction. Option (b) is safer and simpler -- just remove the inner `db.transaction()` wrapper and use the passed-in `db`/`tx` directly.

2. **Should invalid mediaIds return an error?**
   - What we know: `associateMediaToPost` silently skips mediaIds where `postId IS NOT NULL` or the id doesn't exist.
   - What's unclear: Whether the frontend expects a success response even when some media items couldn't be attached.
   - Recommendation: Add a count check after association -- if fewer rows were updated than mediaIds provided, log a warning. Don't block the post creation (media might have been deleted between upload and post creation).
