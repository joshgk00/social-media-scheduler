## Page Contracts

### `/setup` -- First-Time Setup Wizard

**Layout:** Vertically centered card (max-width 400px) on a full-bleed `--background` surface. No navigation chrome. The card has a simple heading and the form.

**Form fields (single form, not multi-step):**
1. Email (type="email", required)
2. Password (type="password", required, min 12 chars)
3. Confirm Password (type="password", required, must match)
4. Timezone (searchable select, IANA timezone list, default to browser's `Intl.DateTimeFormat().resolvedOptions().timeZone`)

**Interactions:**
- Password field shows character count below input: "{N} / 12 minimum". Text color is `--muted-foreground` when < 12, `--success` when >= 12.
- Confirm password shows inline error "Passwords do not match" only after the field loses focus and values differ.
- Submit button: "Create Account". Disabled until all validation passes. Shows spinner during submission.
- On success: redirect to `/login` with a transient success toast "Account created. Sign in to continue."
- On error (user already exists / 403): show alert "Account already exists. Sign in instead." with link to `/login`.

**Accessibility:**
- `<main>` landmark wrapping card
- All inputs have `<label>` elements
- Error messages use `aria-describedby` linked to the input
- Submit button uses `aria-busy="true"` during loading

---

### `/login` -- Login Page

**Layout:** Vertically centered card (max-width 400px) on full-bleed `--background`. No navigation chrome. Minimal branding: app name "Social Media Scheduler" as the card heading at Display size.

**Step 1 -- Credentials:**
1. Email (type="email", required)
2. Password (type="password", required)
3. "Sign In" primary button (full width)
4. "Forgot password?" link below button, links to `/recover`

**Interactions:**
- Submit button disabled while fields are empty. Shows spinner during submission.
- On invalid credentials: inline alert below the form "Invalid email or password." (generic, never reveals which field is wrong -- per D-06).
- On rate limit lockout: replace form with alert "Too many failed attempts. Try again in {minutes} minutes." Show countdown if feasible, otherwise static message.
- On success (no 2FA): redirect to original URL or `/` (dashboard placeholder).
- On success (2FA enabled): transition to Step 2.

**Step 2 -- TOTP Code (conditional, same card):**
- Heading changes to "Two-Factor Authentication"
- Body text: "Enter the 6-digit code from your authenticator app."
- Single input: 6-digit numeric code, `inputMode="numeric"`, `pattern="[0-9]{6}"`, `autocomplete="one-time-code"`, centered text, 44px height, larger font (20px).
- "Verify" primary button (full width)
- "Back to login" text link below
- 5-minute timeout countdown shown as muted text: "Code expires in {M}:{SS}". On expiry, redirect to Step 1 with toast "Session expired. Please sign in again."
- On invalid code: inline error "Invalid code. Please try again." with remaining attempts shown if rate-limited.

**Accessibility:**
- Step transition announced via `aria-live="polite"` region
- TOTP input has `aria-label="6-digit verification code"`
- Countdown uses `aria-live="off"` (not announced repeatedly)

---

### `/recover` -- Account Recovery

**Layout:** Vertically centered card (max-width 400px) on full-bleed `--background`. No navigation chrome.

**Step 1 -- Email Verification:**
1. Heading: "Account Recovery"
2. Body text: "Answer your security questions to reset your password."
3. Email input (type="email", required) -- confirms identity
4. "Continue" primary button

**Step 2 -- Security Questions (after email verified):**
- Shows all 3 configured security questions as read-only labels with answer inputs below each
- "Verify Answers" primary button
- On failure: generic "Incorrect answers." (no indication which ones are wrong)
- On rate limit: same lockout pattern as login

**Step 3 -- New Password (after questions verified):**
- Heading: "Set New Password"
- New Password input with character count indicator (same as setup)
- Confirm New Password input
- "Reset Password" primary button
- On success: redirect to `/login` with toast "Password reset. 2FA has been disabled. Sign in with your new password." All sessions invalidated per D-16.

**No security questions configured state:** If user has not set up security questions, Step 1 shows an alert: "No recovery method configured. Security questions must be set up in Settings before account recovery is available."

**Accessibility:**
- Each step transition announced via `aria-live="polite"`
- Security question labels are read-only text (not inputs)

---

### `/settings` -- User Settings

**Layout:** Full-width page with a maximum content width of 640px, centered. Page title "Settings" at Display size. Three stacked sections, each in a card component. Per-section "Save" buttons (per D-28).

**Section 1: Profile**
| Field | Type | Notes |
|-------|------|-------|
| Profile Image | Avatar with "Change" overlay on hover, file input (accept="image/*") | Circular, 80px diameter. Shows initials fallback if no image. |
| First Name | Text input | Optional |
| Last Name | Text input | Optional |
| Username | Text input | 3-100 chars, inline validation |
| Email | Text input (type="email") | Required |
| "Save Profile" | Button (secondary style) | Right-aligned within section |

**Section 2: Preferences**
| Field | Type | Notes |
|-------|------|-------|
| Timezone | Searchable select | Full IANA timezone list, grouped by region. Default from setup. |
| Date Format | Select | 8 options (see Copywriting Contract below) |
| Entries Per Page | Select | Options: 10, 25, 50, 100 |
| "Save Preferences" | Button (secondary style) | Right-aligned within section |

**Section 3: Security**

This section displays current status and opens modals for changes.

| Item | Display | Action |
|------|---------|--------|
| Password | "Last changed: {date}" or "Never changed" | "Change Password" button opens modal |
| Two-Factor Auth | Badge: "Enabled" (green) or "Disabled" (muted) | "Set Up 2FA" or "Disable 2FA" button opens modal |
| Security Questions | "{N} of 3 configured" or "Not configured" | "Configure" button opens modal |
| Active Sessions | "You are logged in on {N} device(s)" | "Log Out Other Sessions" destructive button |
| Last Login | "{relative time} ({absolute datetime})" | Display only |

**Interactions:**
- Each section saves independently. On save: success toast "Profile saved." / "Preferences saved." / etc.
- Unsaved changes: if user modifies a field and navigates away without saving, no blocking prompt (single-user tool, low risk).
- Save button disabled when no changes detected (compare to initial loaded values).
- Loading: each section shows skeleton placeholders while data loads.

## Modal Contracts

### Change Password Modal

**Trigger:** "Change Password" button in Security section.

**Fields:**
1. Current Password (type="password", required)
2. New Password (type="password", required, min 12 chars)
3. Confirm New Password (type="password", required)

**Interactions:**
- Character count indicator on New Password field: "{N} / 12 minimum". Same color behavior as setup.
- "Change Password" primary button. Disabled until all validation passes.
- On success: close modal, success toast "Password changed. Other sessions have been signed out." (per D-23)
- On wrong current password: inline error "Current password is incorrect."
- Cancel button to dismiss.

---

### 2FA Setup Modal

**Trigger:** "Set Up 2FA" button in Security section (only shown when 2FA is disabled).

**Content:**
1. Heading: "Enable Two-Factor Authentication"
2. QR code image: prominently displayed, 200x200px, centered. Generated from `otpauth://` URI.
3. Text secret: displayed below QR code in a monospace copyable block with a "Copy" button. Background `--secondary` with `--border` outline. Font: `font-mono`, 14px.
4. Instructions: "Scan the QR code with your authenticator app (Google Authenticator, Authy, etc.) or manually enter the secret key."
5. Verification input: "Enter the 6-digit code from your app to verify setup." Same styling as login TOTP input (44px height, centered, 20px font).
6. "Verify & Enable 2FA" primary button.

**Interactions:**
- QR code generates on modal open (API call to get TOTP secret).
- Copy button: copies secret to clipboard, button text changes to "Copied!" for 2 seconds.
- On valid code: close modal, update badge to "Enabled" (green), success toast "Two-factor authentication enabled."
- On invalid code: inline error "Invalid code. Make sure your authenticator time is synced."
- Cannot close modal by clicking outside while verification code input has focus (prevent accidental dismissal after scanning).

---

### Disable 2FA Modal

**Trigger:** "Disable 2FA" button in Security section (only shown when 2FA is enabled).

**Fields:**
1. Current Password (type="password", required)
2. Current TOTP Code (6-digit input, same styling as login)

**Interactions:**
- "Disable 2FA" destructive button (red).
- Confirmation text above button: "This will remove two-factor authentication from your account."
- On success: close modal, update badge to "Disabled" (muted), success toast "Two-factor authentication disabled."
- On failure: inline error "Invalid password or code."

---

### Security Questions Modal

**Trigger:** "Configure" button in Security section.

**Content:**
3 question-answer pairs:

| Field | Type |
|-------|------|
| Question 1 | Select from predefined list |
| Answer 1 | Text input |
| Question 2 | Select from predefined list (excludes Question 1's selection) |
| Answer 2 | Text input |
| Question 3 | Select from predefined list (excludes Questions 1 and 2's selections) |
| Answer 3 | Text input |

**Predefined Security Questions (10 questions):**
1. What was the name of your first pet?
2. What city were you born in?
3. What was the name of your elementary school?
4. What was your childhood nickname?
5. What is your mother's maiden name?
6. What was the make of your first car?
7. What street did you grow up on?
8. What was the name of your first employer?
9. What is the name of your favorite childhood friend?
10. What was your favorite food as a child?

**Interactions:**
- Each subsequent question select excludes already-selected questions.
- All 3 pairs required to save. "Save Security Questions" primary button.
- If questions already configured: pre-populate question selects (answers are NOT shown -- they're hashed).
- Help text below the form: "Answers are case-insensitive. You will need all 3 correct answers to recover your account."
- On success: close modal, update display to "3 of 3 configured", success toast "Security questions saved."

## Interaction Patterns

### Protected Route Pattern
All routes except `/setup`, `/login`, and `/recover` are protected. Unauthenticated requests redirect to `/login?redirect={encodedOriginalPath}`. After login, redirect to the decoded path or `/` if none.

### Auth State Management
Use TanStack Query for server auth state:
- `GET /api/auth/me` query returns user object or 401. Cached with `staleTime: 5 * 60 * 1000` (5 minutes).
- On 401 from any API call, TanStack Query's global `onError` invalidates the auth query and triggers redirect to `/login`.
- Auth check query runs on app mount; shows full-page skeleton loader until resolved.

Use Zustand for transient UI auth state only:
- `redirectAfterLogin: string | null` -- the path to redirect to after successful login.
- Nothing else. All user data comes from the TanStack Query cache.

### Setup Guard
Before rendering any route, a setup check runs:
- `GET /api/auth/setup-status` returns `{ needsSetup: boolean }`.
- If `needsSetup === true`, redirect all routes to `/setup`.
- This endpoint does not require authentication.

### Form Validation
- All forms use React Hook Form with Zod resolver.
- Validation runs on blur for individual fields, on submit for the full form.
- Server-side errors (401, 403, 409, 429) are caught and displayed as form-level alerts, not field-level errors.

### Toast Behavior
- Position: bottom-right.
- Duration: 5 seconds for success, 8 seconds for errors.
- Max 3 toasts visible at once; older ones auto-dismiss.
- Toasts are dismissible by click.

### Loading States
- Buttons show a spinner icon (Lucide `Loader2` with `animate-spin`) and are disabled during async operations.
- Settings page sections show skeleton placeholders while loading.
- Login card content is static (no loading state needed -- the form is always ready).

### Transitions
- Modal open/close: 150ms ease-out (handled by shadcn Dialog).
- Toast enter: slide in from right, 200ms. Toast exit: fade out, 150ms (handled by shadcn Sonner/Toast).
- No page transition animations for this phase.

## Accessibility Checklist (Phase 2 Specific)

- [ ] All form inputs have visible `<label>` elements (WCAG 1.3.1)
- [ ] Error messages linked via `aria-describedby` on the relevant input (WCAG 1.3.1)
- [ ] Color contrast meets 4.5:1 for all text against `--background` and `--card` surfaces (WCAG 1.4.3)
- [ ] TOTP input and all buttons meet 44px minimum touch target (WCAG 2.5.5)
- [ ] Password visibility toggle (if implemented) has `aria-label` reflecting current state

- [ ] Step transitions in login and recovery announced via `aria-live="polite"` (WCAG 4.1.3)
- [ ] Rate limit countdown does NOT use `aria-live` to avoid repetitive announcements
- [ ] Keyboard navigation: Tab through form fields, Enter to submit, Escape to close modals
